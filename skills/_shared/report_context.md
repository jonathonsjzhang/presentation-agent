# 汇报场景共享规则

所有 Agent 必须读取 `report_context`，并把它作为高优先级约束。`report_context` 由任务定位 Agent 生成，后续 Agent 不得擅自改写；如果发现上下游冲突，只能在本环节 artifact 中标记 `context_conflicts` 或 `open_questions`。

## 三个必填维度

### audience

只能使用以下枚举之一：

- `board`: 董事会
- `exec_office`: 总办汇报
- `strategy_lead`: 战略负责人
- `business_team`: 业务负责人和业务团队
- `external`: 外部分享

### report_type

只能使用以下枚举之一：

- `deep_dive`: 专题汇报
- `quick_sync`: 信息快速同步

### output_format

只能使用以下枚举之一：

- `document`: 文档
- `ppt`: PPT
- `html`: HTML

## 受众差异

### board: 董事会

- 关注：长期价值、战略取舍、重大资源配置、风险收益、治理责任。
- 结论形态：必须落到“做/不做、优先级、投入边界、风险是否可接受”。
- 证据要求：强结论必须强证据；口径、时间窗、对比基准必须清楚。
- 语气：克制、审慎、专业，不用命令式和营销式表达。
- 页面密度：少页高密度，主线页不陷入执行细节。
- 高风险错误：把运营动作当战略建议；用单一案例支撑重大决策；缺少风险和反方。

### exec_office: 总办汇报

- 关注：跨团队协调、关键阻塞、阶段性进展、需要拍板的问题。
- 结论形态：必须明确“当前判断、卡点、选项、需要谁决定什么”。
- 证据要求：足以支持会议推进，不必展开成完整行业论文。
- 语气：直接、可决策、节奏快。
- 页面密度：中高，背景只保留必要上下文。
- 高风险错误：只讲背景不讲请求；只提问题不提选项；不说明责任边界。

### strategy_lead: 战略负责人

- 关注：问题定义、分析框架、关键假设、反证、竞争格局、可验证路径。
- 结论形态：必须说明“判断如何推出、什么条件下成立、怎样验证或推翻”。
- 证据要求：允许复杂推理，但必须显式写出前提和边界。
- 语气：分析型、严谨、可讨论。
- 页面密度：高，保留推导链和替代解释。
- 高风险错误：框架漂亮但不产出洞察；维度混搭；只给答案不交代推理。

### business_team: 业务负责人和业务团队

- 关注：业务影响、指标变化、场景机会、优先级、资源与落地动作。
- 结论形态：必须说明“业务上先做什么、不做什么、为什么、看什么指标”。
- 证据要求：指标和场景要清楚，能转化为动作。
- 语气：务实、清楚、有行动感。
- 页面密度：中高，允许必要执行细节。
- 高风险错误：只停留在董事会式抽象判断；没有动作、责任边界和优先级。

### external: 外部分享

- 关注：行业启发、洞察表达、可信案例、可传播记忆点。
- 结论形态：必须形成听众可带走的新认知，不输出内部决策指令。
- 证据要求：公开可解释，敏感数据必须脱敏或泛化。
- 语气：开放、清楚、有故事性。
- 页面密度：中等，术语首次出现要解释。
- 高风险错误：泄露内部口径；把内部行动建议公开化；堆砌术语导致听众无法理解。

## 汇报性质差异

### deep_dive: 专题汇报

- 目标：形成判断、解释原因、提出选择或行动方向。
- 推荐结构：问题定义 -> 一句话结论 -> 分析框架 -> 关键论证 -> 影响/建议 -> 风险/Q&A。
- 质量重心：结构化、洞察深度、证据充分、故事线、可决策性。
- 最低标准：必须有主论点、2-4 个分论点、关键反方和边界条件。

### quick_sync: 信息快速同步

- 目标：快速让听众知道发生了什么、为什么重要、下一步看什么。
- 推荐结构：一句话结论 -> 关键事实 -> 影响判断 -> 后续动作/待观察。
- 质量重心：短、准、清楚、有下一步。
- 最低标准：不能强行拔高成大战略，不能塞入不必要的长推理。

## 材料格式差异

### document: 文档

- 适合：完整推理、留档、细读、多人异步审阅。
- 要求：标题层级清楚，段落能独立阅读，保留证据来源和推理过程。
- 重点：逻辑完整性和可追溯性。
- 禁忌：PPT 式碎片化，读者离开讲解者就看不懂。

### ppt: PPT

- 适合：会议讲解、现场决策、口头汇报。
- 要求：一页一结论，标题即结论，图表/证据服务标题，页面之间形成故事线。
- 重点：页面焦点、标题串联、讲解动线。
- 禁忌：大段文字、无结论标题、信息页互不相干。

### html: HTML

- 适合：交互浏览、长短结合、可展开证据、轻量发布。
- 要求：摘要区、导航、锚点、可折叠证据/附录、移动端可读。
- 重点：信息架构和浏览路径。
- 禁忌：把 PPT 逐页搬成网页，或把长文堆成无导航页面。

## 全局硬约束

- 不得编造数据、来源、访谈或案例。
- 缺口必须显式标记，不得用漂亮表达掩盖。
- 强建议必须有强证据和边界条件。
- 每个 Agent 输出都必须能被下游按字段读取，不依赖猜测。
- 如果任务定位不完整，后续 Agent 必须降级处理，并把缺口写入 `open_questions` 或 `data_gaps`。
