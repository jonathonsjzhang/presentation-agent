# 汇报助手系统简化设计方案

> 状态：v0.4 实施基线，后续可继续完善 Worker 方法论与模板
> 建议版本：`v0.4`
> 日期：2026-07-15
> 核心目标：让脚手架重新服务于战略分析与报告生产，而不是让 Worker 主要精力用于维护中间 JSON、校验和返工状态。

## 0. 结论与已确认决策

本轮不是继续修补旧版复杂交接协议，而是主动收缩系统边界。目标主链保留五个专业 Worker：

```text
Analysis → Storyline → Report → QA → Format
```

但五个 Worker 不再通过逐层扩张的复杂 JSON schema 交接语义内容。新的基本原则是：

> Brief 是所有 Worker 共享的稳定项目上下文；Analysis、Storyline、Report 和 QA 以规范化完整文本交接；JSON 只用于机器真正需要读取的控制信息。

本轮已确认以下决策：

| ID | 决策 |
|---|---|
| D1 | Brief 保持结构化，并在每个 Worker 的输入中完整传递。 |
| D2 | Analysis 输出一份完整、规范、可独立阅读的分析文档，不再拆成大量 JSON 字段。 |
| D3 | Storyline 输出一份完整、规范、可独立阅读的故事线文档，不再把 core answer、sections、visual plan 拆成复杂对象。 |
| D4 | Report 接收 Brief、完整 Analysis、完整 Storyline 和可回查 Evidence，直接输出 canonical `report.md`。 |
| D5 | QA 保留为独立 Worker，避免把提问视角额外压给 Report；但 QA 自身流程和产物必须简化。 |
| D6 | 首次生产默认经过 QA；Report 或 Format 的局部返工默认可以跳过 QA。 |
| D7 | 删除跨阶段视觉台账传递。Format 直接基于报告、证据资产和 renderer 能力完成可视化。 |
| D8 | 只保留三类校验：输入输出完整性、Analysis/Storyline 人工 Gate、最终真实渲染检查。 |
| D9 | 返工按明确阶段执行，不再由 artifact 中的 `target_agent` 自动决定责任人。 |
| D10 | Memory 继续保留，但只作为生成前的软性经验，不参与硬路由和多层阻塞检查。 |

---

## 1. 为什么需要简化

### 1.1 当前问题不是单个 Schema 太长，而是同一语义被重复建模

旧版协议中，一项可视化论据会沿多个 Artifact 重复表达：

```text
Analysis.visual_evidence_candidates
  → Storyline.visual_evidence_plan
  → Report.visual_evidence_placements
  → QA 原样复制 visual_evidence_placements
  → Format.visuals
  → visual evidence audit
  → cross-stage review
  → renderer
```

同一个事实需要在不同阶段保持 ID、标题、位置、类型、来源、必需性和数据结构一致。任何一次自然语言改写都可能被判为接口错误。系统因此把大量执行成本花在同步中间状态，而不是提高分析和写作质量。

### 1.2 内容工作被编码逻辑切碎

Analysis 和 Storyline 本质上是完整的思考成果：它们需要呈现观点如何形成、证据如何支持、有哪些反向解释、主线如何递进。过度拆成字段会产生三个问题：

1. 下游 Worker 看到的是碎片，而不是完整思路；
2. Worker 为满足字段而重复或压缩表达；
3. Schema 合规被误认为语义完整。

新的设计应让下游 Worker 像接手一位分析师的工作底稿一样，先读到一份结构清楚、上下文完整的文档，再继续工作。

### 1.3 多套校验和返工机制形成了第二套业务

当前单个 Worker 产出可能依次经过：

```text
Schema gate
→ Worker review
→ stop check
→ Manager acceptance
→ cross-stage review
→ visual evidence audit
→ renderer validation
→ final gate
```

不同检查器可能持有不同事实源，并产生相互矛盾的责任归属。返工又会被固定链路和 runtime normalization 放大成整链回放。

本轮简化的重点不是把这些检查器做得更聪明，而是删除大部分中间检查，只在真正不可替代的位置保留约束。

## 2. 设计原则

### 2.1 内容文档优先

完整观点、故事线、正文和追问使用 Markdown 表达。文本应：

- 可独立阅读；
- 有稳定章节模板；
- 保留论证上下文；
- 允许正常改写标题和表述；
- 可通过 Evidence ID 回查来源；
- 不要求下游逐字段重新组装语义。

### 2.2 JSON 只用于机器边界

严格 JSON 只保留在以下场景：

- Brief 的稳定项目设置；
- Evidence Catalog 和数据资产索引；
- runtime 的任务状态与 Artifact 路径；
- Renderer 的真实输出清单；
- 明确的错误代码和执行状态。

不再用 JSON 承载章节论证、观点展开、故事线逻辑和视觉位置等主要语义内容。

### 2.3 单一事实源，不做多份语义投影

| 信息 | 唯一事实源 |
|---|---|
| 项目目标、受众、篇幅、限制 | `brief.json` |
| 原始材料及可回查来源 | `evidence_catalog.json` / data assets |
| 分析判断 | `analysis.md` |
| 已批准故事线 | `storyline.md` |
| 正式正文 | `report.md` |
| 听众追问 | `qa.md` |
| 最终视觉和排版结果 | 真实交付文件 + `render_manifest.json` |

任何信息不在多个 Artifact 中复制维护。下游如需使用，直接读取事实源。

### 2.4 Happy path 简单，返工路径显式

首次生产可以保持清晰的默认顺序；动态能力主要用于明确的返工和人工 Gate，而不是让 runtime 猜测任意任务图。

### 2.5 先阶段级返工，不做字段级依赖引擎

v0.4 暂不建设复杂的字段级 DAG、语义 fingerprint 或自动失效传播。先把返工做到阶段级、可解释、可预测，已经可以解决大多数整链回放问题。

已实现的阶段级控制面包括 `report revise --stage ...` 和 `report continue` 的 until-boundary 语义；后者只消费已存在的 Worker 输出，不生成专业内容。

## 3. 目标架构

### 3.1 首次生产主链

```mermaid
flowchart LR
    M["Raw Materials"] --> E["Evidence Intake"]
    E --> B["Brief Gate"]
    B --> A["Analysis"]
    A --> AG["Analysis Human Gate"]
    AG --> S["Storyline"]
    S --> SG["Storyline Human Gate"]
    SG --> R["Report"]
    R --> Q["QA"]
    Q --> F["Format"]
    F --> V["Final Render Check"]
    V --> H["Final Human Gate"]
```

Evidence Intake 是 run-level 输入准备任务，不参与后续语义返工链。它只负责材料读取、索引和漏读报告。

### 3.2 所有 Worker 的共同输入

每个 Worker 都接收完整 Brief，而不是只接收 runtime 投影出的少数字段：

```text
brief.json
当前 Worker 所需的上游 canonical 文档
evidence_catalog.json 或其索引入口
本轮用户 revision feedback（如有）
与当前 Worker 相关的少量 memory（如有）
```

Brief 的建议最小结构：

```json
{
  "topic": "报告主题",
  "research_purpose": "本项目希望回答的问题",
  "research_direction": "当前研究假设或讨论方向",
  "audience": "汇报对象",
  "project_type": "分析类或梳理类",
  "delivery_target": "document | ppt | html",
  "report_length": "篇幅要求",
  "high_confidence_evidence": ["EV-001"],
  "constraints": ["口径、保密、风格和边界要求"],
  "user_intent": "用户原始意图的简明记录"
}
```

Brief 在用户确认后成为稳定事实源。中间 Worker 不得自行修改；若项目目标变化，由 Manager 显式更新并记录新版本。

## 4. 五个 Worker 的简化职责与产物

### 4.1 Analysis：输出完整分析文档

#### 输入

- `brief.json`
- `evidence_catalog.json`
- 原始数据资产的受控访问入口
- Analysis memory（少量软提示）

#### 输出

```text
analysis.md
```

Analysis 不再输出 `findings[]`、`thesis_options[]`、`visual_evidence_candidates[]` 等复杂 JSON。它输出一份完整分析底稿，建议使用以下规范模板：

```markdown
# Analysis

## 1. 研究问题与材料边界
说明本次分析试图回答什么、读了哪些材料、哪些材料未能覆盖。

## 2. 核心观察与事实
按主题说明材料中最重要的事实、变化和差异，并用 [EV-xxx] 标注来源。

## 3. 完整观点与解释
展开主要判断、形成机制、不同证据之间的关系，以及这些事实意味着什么。

## 4. 竞争性解释与反向证据
说明替代解释、反例、口径限制和不能过度推断的部分。

## 5. 候选主论点
### 方案 A：……
完整说明主论点、支撑逻辑、适用场景和主要代价。

### 方案 B：……
……

## 6. 建议讨论的问题
列出需要用户选择或判断的关键分歧。
```

模板用于保证文本规范和信息完整，不用于把内容重新变成字段。Evidence ID 只承担追溯功能，不承担跨阶段数据库主键功能。

#### Human Gate

Manager 将 Analysis 中的候选主论点和必要说明展示给用户。用户确认后，把选择结果作为一段简短的批准记录附在 `analysis_approval.md`，Storyline 同时读取完整 `analysis.md` 和批准记录。

### 4.2 Storyline：输出完整故事线文档

#### 输入

- `brief.json`
- `analysis.md`
- `analysis_approval.md`
- `evidence_catalog.json` 回查入口
- Storyline memory

#### 输出

```text
storyline.md
```

Storyline 不再输出 `core_answer`、`sections[]`、`visual_evidence_plan[]` 等复杂 JSON，而是形成一份完整、可讨论、可直接交给写作者的故事线文档：

```markdown
# Storyline

## 1. 核心回答
用一段完整文字回答本次汇报最终要说什么。

## 2. Executive Summary 草案
说明最重要的结论、依据、边界和对听众的意义。

## 3. 整体叙事逻辑
解释为什么按照这一顺序展开，各章节如何共同证明核心回答。

## 4. 章节故事线

### 第一章：章节结论式标题
- 本章要回答的问题
- 核心论证
- 需要使用的关键事实与论据：[EV-xxx]
- 与上一章、下一章的关系
- 不应越界的判断

### 第二章：……
……

## 5. 结尾与讨论方向
说明报告如何收束，以及希望听众形成什么认识或讨论。

## 6. 取舍与边界
说明哪些相关信息不进入主线、哪些内容留作附录或追问。
```

Storyline 文档必须表达完整思路，而不是成为 Report Worker 需要再次拼装的提纲字段集合。

#### Human Gate

用户确认的是完整故事线，而不是某些 JSON 字段。批准记录写入 `storyline_approval.md`，之后 Report 将 `storyline.md` 视为已批准的写作蓝图。

### 4.3 Report：只负责写成正式报告

#### 输入

- `brief.json`
- `analysis.md`
- `analysis_approval.md`
- `storyline.md`
- `storyline_approval.md`
- `evidence_catalog.json` 和数据资产回查入口
- Report memory
- 本轮 revision feedback（如有）

#### 输出

```text
report.md
```

Report 不再输出视觉位置清单，也不需要复制 Storyline 或 Evidence 的结构字段。其任务是把已确认的观点与故事线写成一篇完整、严肃、可独立阅读的 Markdown 报告。

Report 可以在正文中自然使用 Evidence 引用，例如 `[EV-012]`，但不需要写 `[可视化论据：VE-01]` 等机器位置标记。是否可视化、使用何种图表、插入在哪里，由 Format 基于正文结构和证据资产判断。

Report 只执行一次同上下文自检：核对是否忠实于 Brief 和 Storyline、是否有无来源数字、是否遗漏关键 caveat、是否明显超出篇幅。可在当前 Worker 内直接修正，不产生独立 Reviewer 任务和 Manager 返工轮次。

### 4.4 QA：保留独立 Agent，但只输出追问文档

#### 为什么保留

QA 需要切换到听众、决策者或挑战者视角。把这项工作并入 Report 会让 Report 同时承担写作者和反方提问者两种认知负担，因此继续保留独立 Worker。

#### 输入

- `brief.json`
- `report.md`
- `storyline.md`，用于理解主线和边界
- `evidence_catalog.json` 回查入口
- QA memory

#### 输出

```text
qa.md
```

QA 不再复制整份 `report_markdown`，也不复制任何视觉位置。建议模板：

```markdown
# 听众可能追问的问题

1. ……？
   - 追问缘由：该问题挑战哪一项核心判断或证据边界。

2. ……？
   - 追问缘由：……
```

QA 只负责问题，不写答案，不改 Report，不承担跨阶段一致性字段。问题数量由 Brief 的场景和篇幅决定，不再默认强制 6—10 个。

#### QA 的运行与跳过规则

首次生产默认运行一次 QA。后续返工遵循：

- 只改 Report 措辞、篇幅或局部结构：跳过 QA，复用既有 `qa.md`；
- 只改 Format：跳过 QA；
- 用户明确修改追问：只重跑 QA；
- Storyline 或核心论点发生实质变化：QA 随 Report 之后重跑；
- 没有可复用 QA 且本次走快速返工：允许 Format 不附加 QA，不得因此阻塞交付。

Format 读取 `report.md` 和可选的 `qa.md`。是否把 QA 作为报告末尾章节、独立附件或不纳入当前载体，由 Brief 和载体规则决定，不再通过 QA 重写 Report Artifact 实现。

### 4.5 Format：直接对 canonical 文档做载体化

#### 输入

- `brief.json`
- `report.md`
- 可选 `qa.md`
- `evidence_catalog.json`
- 可用的数据资产和图片资产
- 目标 renderer 的能力声明
- Format memory

#### 输出

- DOCX、PPTX 或 HTML 真实文件；
- 由 runtime 生成的最小 `render_manifest.json`。

Format Worker 不再生成一套需要跨阶段对齐的 `formatted_material.v2.visuals[]`。它应基于 Report 的实际 Markdown 结构决定信息层级和视觉表达，只允许使用 renderer 已声明支持的能力。

`render_manifest.json` 不是语义产物，只记录机器事实：

```json
{
  "status": "success",
  "deliverable_path": ".../report.docx",
  "page_count": 7,
  "generated_assets": [".../chart-01.png"],
  "degraded_assets": [],
  "warnings": []
}
```

这一 Manifest 由 renderer/runtime 生成，不能依赖模型自报成功。

## 5. 最小运行协议

### 5.1 内容 Artifact 与运行信封分离

Worker 的专业产物是 Markdown 或真实文件；runtime 只需要一个极小的执行信封：

```json
{
  "status": "done | blocked",
  "artifact_path": ".../analysis.md",
  "issues": []
}
```

信封只回答三个问题：任务是否完成、产物在哪里、是否存在无法继续的阻塞。它不承载专业内容，不包含下一责任 Worker，也不复制 Artifact。

### 5.2 最小 Task Packet

```json
{
  "agent_id": "report",
  "objective": "基于已批准故事线写成正式报告",
  "brief_path": ".../brief.json",
  "artifact_paths": [
    ".../analysis.md",
    ".../storyline.md",
    ".../evidence_catalog.json"
  ],
  "revision_feedback": "可选的用户原话"
}
```

runtime 只验证路径存在和 Worker 类型合法，不对语义内容做复杂 normalization。

### 5.3 Manager 的简化职责

Manager 保留以下职责：

- 汇总并确认 Brief；
- 展示 Analysis 候选主论点并接收用户决定；
- 展示 Storyline 并接收用户决定；
- 根据用户反馈明确选择返工阶段；
- 在 Worker 返回 `blocked` 时决定询问用户还是指定上游返工；
- 展示最终材料并接收修改意见。

Manager 不再：

- 为每个正常 Worker 手写 acceptance JSON；
- 根据中间 Artifact 的 `target_agent` 自动决定责任方；
- 重新描述 cross-stage reviewer 已经检查的内容；
- 为固定 happy path 重复生成下一阶段 task packet；
- 修改 Worker 的专业产物。

正常情况下，Report 和 QA 完成最小文件校验后自动进入下一阶段，不经过单独 Manager acceptance。

## 6. 校验机制：只保留三层

### 6.1 第一层：输入输出完整性

由 runtime 确定性检查：

- 输入路径存在；
- 文件非空且可读取；
- Brief 可解析；
- Evidence Intake 扫描到的每个源文件都有 `parsed / unsupported / failed / skipped` 状态；
- Markdown 至少包含约定的一级标题和必要章节；
- Evidence 引用如果出现，必须能在 Catalog 中解析；
- Worker 输出时间晚于本轮任务开始时间；
- Renderer 声明的交付文件真实存在。

可在当前 Worker 内修复的小问题，只允许原地重试一次，不进入跨阶段 Manager 返工。

### 6.2 第二层：两个语义 Human Gate

只保留：

1. Analysis 主论点确认；
2. Storyline 确认。

这两个 Gate 涉及真正的战略判断和取舍，值得由用户拍板。Report、QA 不再增加默认人工 Gate。

### 6.3 第三层：最终真实渲染检查

Format 后只检查真实交付结果：

- 文件是否生成；
- 最终页数；
- 中文字体和字符是否正常；
- 页面是否空白、溢出或不可读；
- 声明生成的图片是否真实存在；
- 是否发生静默降级；
- 图表是否使用真实 Evidence 数据；
- DOCX/PPT/HTML 是否能够打开和完成截图。

生产链不再比较 Storyline 标题、Report 标题和 Format `section_heading` 是否逐字一致。

## 7. 简化返工机制

### 7.1 返工指令

Manager 输出明确的阶段级指令：

```json
{
  "action": "revise",
  "stage": "report",
  "feedback": "正文过长，压缩第三节，不改变已确认故事线"
}
```

`stage` 只允许：

```text
analysis | storyline | report | qa | format
```

用户明确指定阶段时，以用户决定为准；用户只给自然语言反馈时，由 Manager 分类，但 runtime 不得覆盖 Manager 的阶段选择。

### 7.2 返工路径

| 返工阶段 | 默认后续路径 | Human Gate |
|---|---|---|
| Analysis | Analysis → Storyline → Report → QA → Format | 重新经过 Analysis 和 Storyline Gate |
| Storyline | Storyline → Report → QA → Format | 重新经过 Storyline Gate |
| Report | Report → Format | 默认跳过并复用既有 QA |
| QA | QA → Format | 无中间 Gate |
| Format | Format | 直接回最终检查 |

若 Report 返工明确改变核心结论或故事线，Manager 应把返工目标提升为 Storyline，而不是在 Report 路径上自动猜测语义失效。

### 7.3 熔断

相同阶段、相同错误连续出现两次时：

- 停止自动重试；
- 保留当前产物；
- 向 Manager 返回明确错误和诊断文件；
- 不自动切换责任 Worker；
- 不整链回放。

runtime 以 `stage + error_code + stable_detail` 生成稳定签名。第一次同类失败在同一 task 中进入 revise；第二次将状态置为 `blocked_repeated_failure`，并把 `runtime_failure.v1` 返回人工 Gate。

## 8. 重新判断：哪些能力仍然有必要保留

### 8.1 必须保留

| 能力 | 保留原因 | 简化方式 |
|---|---|---|
| Brief Gate | 项目目的、受众和边界是所有工作的共同约束 | 一份结构化 Brief，所有 Worker 完整读取 |
| Evidence Intake | 防止漏读材料，并支持事实回查 | 只做解析、索引和状态报告，不做观点判断 |
| Evidence ID 与数据资产引用 | 防止无来源数字和编造图表 | 只用于引用事实源，不用于章节和视觉位置同步 |
| Analysis Human Gate | 主论点选择属于高价值判断 | 用户读完整 Analysis 后确认 |
| Storyline Human Gate | 论证主线决定整篇报告 | 用户读完整 Storyline 后确认 |
| Worker Skill | 保留专业分工、核心准则、完整 Workflow 与质量方法 | 不删减方法论主体，只在末尾增加 v0.4 交接载体说明；后续可独立迭代完善 |
| 独立 Worker 上下文 | Analysis、Storyline、Report、QA、Format 的认知任务不同 | 继续使用宿主原生 sub-agent |
| Report canonical Markdown | 保证内容有唯一事实源 | Report 只写 `report.md` |
| QA 独立 Agent | 保留听众挑战视角，避免加重 Report | 只写 `qa.md`，不复制正文 |
| Renderer capability | 防止 Worker设计无法落地的视觉 | Format 只读取实际支持能力 |
| 最终真实渲染检查 | 用户最终消费的是文件，不是中间 JSON | 用真实文件、页面和资产验收 |
| Stage-level revision | 用户反馈必须能局部修改 | 明确 `stage + feedback`，不自动归一化 |
| Learning log / memory | 系统需要从真实反馈中持续学习 | 仅软提示，默认不阻塞、不路由 |
| Doctor | 提前发现解析、字体和渲染环境问题 | 改为真实端到端冒烟测试 |

### 8.2 降级为可选

| 能力 | 新策略 |
|---|---|
| 过程 Reviewer sub-agent | 删除；Worker 自检，runtime 做确定性验证 |
| P0/P1 rubric review | Worker 内同上下文自检；只将确定性问题设为硬阻塞 |
| 多候选生成 | Analysis 可按 Brief 需要给出候选；其他 Worker 默认单解 |
| QA 重跑 | 首次默认运行，局部 Report/Format 返工默认跳过 |
| Page budget 预估 | Report 只做字数级提醒；真实页数以 Format 最终模板为准 |
| Memory promotion | 改为人工或离线维护，不在生产 run 中自动改 rubric |

### 8.3 删除

| 删除项 | 原因 |
|---|---|
| `visual_evidence_candidates` | 让 Analysis 过早承担 Format 的表达决策 |
| `visual_evidence_plan` | 重复维护同一视觉语义 |
| `visual_evidence_placements` | 位置字符串成为脆弱的跨阶段主键 |
| QA 复制 `report_markdown` | 产生第二份正文事实源 |
| QA 复制视觉位置 | 与 QA 专业职责无关 |
| Format `section_heading` 精确匹配 | 自然标题改写不应成为 P0 |
| Storyline 标题逐字保留检查 | 应检查语义忠实，而不是字面一致 |
| `upstream_revision_requests.target_agent` | Artifact 不应拥有调度权 |
| runtime 覆盖 Manager 返工阶段 | 造成错误路由和死循环 |
| 每阶段 Manager acceptance JSON | 固定 happy path 中没有额外判断价值 |
| 多套 cross-stage / visual audit 阻塞器 | 事实源冲突且重复验收 |
| Worker 小字段错误触发完整 respawn | 小问题应原地修复 |
| 字符数边界触发整轮内容返工 | 确定性格式问题不应升级为语义重写 |
| 默认全阶段独立 Reviewer | 成本高，且当前主要瓶颈不是主观质量审查不足 |

## 9. Memory 在简化架构中的位置

Memory 是系统长期演进能力，应保留，但需要退出控制面。

新的生效方式：

```text
用户反馈 / 成功案例
  → learning log 忠实记录
  → 提炼为所属 Worker 的短 memory
  → 下次生成前按相关性取少量提示
```

Memory 不再：

- 参与返工阶段路由；
- 在每个 Worker 完成后全库扫描并产生阻塞；
- 自动生成新的 runtime 硬约束；
- 在生产 run 中自动晋升 rubric；
- 覆盖 Brief 和用户本轮明确要求。

高频经验需要升级为 Skill 或硬校验时，进入独立维护流程，由开发者确认后修改。这样可以避免 Memory 把偶发反馈固化成新的脚手架复杂度。

建议每个 run 保持清楚、扁平的 canonical 目录：

```text
runs/<run_id>/
├── brief.json
├── evidence/
│   ├── evidence_catalog.json
│   ├── data_assets/
│   └── intake_manifest.json
├── analysis/
│   ├── analysis.md
│   └── approval.md
├── storyline/
│   ├── storyline.md
│   └── approval.md
├── report/
│   └── report.md
├── qa/
│   └── qa.md
├── format/
│   ├── deliverable.docx
│   ├── assets/
│   └── render_manifest.json
├── manager_state.json

返工版本可以保存在阶段目录下：

```

runtime 状态只保存当前 canonical 路径、版本、批准状态和当前阶段，不复制正文内容。

## 11. 实施顺序

### Phase 1：先简化 Artifact

1. 新增 `analysis.md`、`storyline.md`、`report.md`、`qa.md` canonical artifact；
2. 修改 Worker Skill，改用规范 Markdown 模板；
3. Report 删除 `visual_evidence_placements`；
4. QA 改为只输出 `qa.md`；
5. Format 直接读取 Report、QA 和 Evidence Assets。

### Phase 2：简化校验和自动推进

1. 删除标题逐字一致检查；
2. 删除跨阶段视觉台账审计；
3. Worker 完成后只做文件和引用完整性检查；
4. Analysis、Storyline之外自动推进；
5. 最终只验真实渲染结果。

### Phase 3：简化 Manager 和返工

1. Manager decision 改为 `action + stage + feedback`；
2. runtime 停止覆盖 Manager 的 revise stage；
3. 实现五条明确返工路径；
4. Report/Format 局部返工默认跳过 QA；
5. 相同错误两次熔断。

### Phase 4：环境与测试收口

1. doctor 增加 PDF、DOCX、中文字体和图表冒烟测试；
2. Evidence Intake 对所有源文件显式报告状态；
3. 用已有四个真实案例做回归；
4. 对比 Worker 次数、人工握手次数、最终完成率和内容质量。

## 12. 验收指标

简化不是以“删除了多少代码”为成功标准，而是以真实流程是否更顺畅为准。

建议至少跟踪：

| 指标 | 目标 |
|---|---|
| 首次完整生产 Worker 数 | 正常为 5 次，不因确定性字段问题额外增加 |
| Report 局部返工 Worker 数 | 2 次：Report + Format |
| Format 局部返工 Worker 数 | 1 次：Format |
| 固定人工 Gate | 3 个：Brief、Analysis、Storyline；最终交付另有最终确认 |
| 跨阶段标题字符串 P0 | 0 |
| Renderer 不支持类型导致的末端失败 | 0；能力不足应在 Format 内立即暴露 |
| Evidence 静默漏文件 | 0 |
| 相同错误连续重派 | 最多 2 次，之后熔断 |
| QA 重复重跑 | Report/Format 局部修改时为 0 |
| 最终文件假成功 | 0；成功必须有真实文件与 render manifest |

同时用人工盲评确认：简化后的 Analysis 和 Storyline 是否比字段化 Artifact 更完整、更容易被下游理解，最终报告质量不能因减少 Schema 而下降。

---

## 13. 本轮非目标

为避免再次过度设计，v0.4 暂不建设：

- 任意 Worker 动态任务图；
- 字段级依赖 DAG；
- LLM 自动判断语义 fingerprint；
- 自动合并多份 Artifact；
- 自动推断复杂的上游责任归属；
- 生产 run 中自动修改 Skill 或 rubric；
- 为每种视觉建立新的跨阶段协议。

如果未来真实案例反复证明某项能力必要，再从最小实现开始增加，而不是预先设计完整框架。

## 14. 一句话总结

> v0.4 的目标不是做一个更聪明的流程控制器，而是让 Analysis、Storyline、Report、QA 和 Format 各自把专业工作做好；Brief 和 Evidence 提供稳定事实上下文，Markdown 承载完整思考，runtime 只负责把文件、Gate、返工和真实交付跑顺。
