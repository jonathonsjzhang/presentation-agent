# 汇报助手插件化打包部署执行方案

## 1. 目标

把当前汇报助手从“只能在本地项目里手动运行的 harness”，升级为一套可被他人安装、更新、使用的插件化工具。

目标用户不是工程师，而是能在 Codex、Claude Code、WorkBuddy 等 Agent 终端里发自然语言指令的人。因此最终体验应该是：

```text
用户：安装汇报助手
Agent：自动 clone / 更新 GitHub 仓库，初始化本地工作区，检查运行环境

用户：帮我做一份给集团战略负责人的汇报
Agent：自动调用汇报助手 harness，按 7-Agent loop 推进，阶段性让用户 review

用户：更新汇报助手
Agent：自动 git pull 官方仓库，迁移 workspace，不覆盖用户历史 memory / runs
```

核心原则：

- 用户不需要理解 git、Python、路径、虚拟环境或 CLI 参数。
- Agent 终端可以代替用户执行安装、更新和调度。
- 官方脚手架代码和用户数据必须分离。
- 接入 skill 要尽量薄，不写死本地路径，不复制复杂内部协议。
- 先用 `skill + git + CLI` 做 MVP，后续再升级到 MCP。

## 2. 推荐总体架构

### 2.1 三层结构

```text
Host Agent Terminal
Codex / Claude Code / WorkBuddy
        |
        | thin report-builder skill
        v
Presentation Agent Repo
runtime / configs / skills / templates / CLI
        |
        | --workspace
        v
User Workspace
memory / runs / artifacts / user config
```

三层职责：

| 层级 | 作用 | 是否随官方更新变化 |
|---|---|---|
| Host Adapter Skill | 告诉 Codex / Claude Code / WorkBuddy 如何安装、更新、调用工具 | 是 |
| Official Repo | 存放 runtime、7-Agent skills、configs、CLI、模板 | 是 |
| User Workspace | 存放用户 memory、runs、artifacts、私有配置 | 否 |

### 2.2 本地目录建议

默认安装到用户 home 目录：

```text
~/PresentationAgent/
├── repo/                         # 官方 GitHub 仓库 clone 下来
├── .venv/                        # 可选虚拟环境
└── workspaces/
    └── default/
        ├── config.toml
        ├── data/
        ├── runs/
        ├── artifacts/
        └── logs/
```

关键约束：

- `repo/` 可以 `git pull` 更新。
- `workspaces/` 只属于用户，不被 `git pull` 覆盖。
- memory、learning log、run state、artifacts 都写入 workspace。
- 官方 repo 只放模板和默认配置，不直接写用户状态。

## 3. 三个阶段的关系

这三个阶段不是替代关系，而是递进关系：

```text
第一阶段：系统可搬家
→ 第二阶段：Agent 终端可调度
→ 第三阶段：调度方式标准化
```

### 第一阶段：CLI 和 workspace 可移植化

目标：让系统不依赖当前开发者电脑路径，可以在别人电脑上 clone 后运行。

重点解决：

- 去掉 `HARNESS_ROOT=/Users/...` 这类硬编码。
- 增加 `--workspace` 参数。
- 把用户数据从 repo 目录中剥离。
- 增加安装、诊断、初始化、更新命令。

### 第二阶段：薄 Host Adapter Skill

目标：让 Codex、Claude Code、WorkBuddy 能通过一个简短 skill 自动安装和调度汇报助手。

重点解决：

- skill 不再写死路径。
- skill 不再展开所有内部 step 细节。
- skill 只负责让 Agent 调用 CLI，并把 CLI 返回的 instruction / present_to_user 呈现给用户。

### 第三阶段：MCP 工具化

目标：把 CLI 调用升级成标准工具接口，让支持 MCP 的 Agent 终端可以直接调用工具。

重点解决：

- 不依赖 shell 命令字符串。
- 暴露 `report_start`、`report_next`、`report_submit`、`record_feedback` 等工具。
- 更适合长期产品化，但不是当前 MVP 必需项。

## 4. 第一阶段执行方案：CLI 和 Workspace 可移植化

### 4.1 新增 workspace 解析机制

新增模块建议：

```text
presentation_agent/workspace.py
```

职责：

- 解析当前 workspace 路径。
- 创建 workspace 目录结构。
- 读取 workspace 配置。
- 提供 data / runs / artifacts / logs 的统一路径。

路径解析优先级：

```text
1. CLI 参数 --workspace
2. 环境变量 PRESENTATION_AGENT_WORKSPACE
3. 当前目录向上查找 .presentation-agent/
4. 默认 ~/PresentationAgent/workspaces/default
```

需要实现的函数：

```python
resolve_workspace(path: str | None) -> Workspace
init_workspace(path: str, template: str | None = None) -> Workspace
find_workspace(start: Path) -> Workspace | None
```

验收标准：

- 不传参数时可以使用默认 workspace。
- 传 `--workspace /tmp/pa-test` 时所有运行数据都写入该目录。
- repo 里的 `data/`、`artifacts/` 不再作为默认用户状态目录。

### 4.2 增加初始化命令

新增 CLI：

```bash
presentation-agent init-workspace --workspace ~/PresentationAgent/workspaces/default
```

作用：

- 创建 workspace 目录。
- 写入 `config.toml`。
- 初始化 `data/global/state.json`。
- 初始化每个 agent 的 memory 文件。
- 检查必要目录是否存在。

建议目录：

```text
workspace/
├── config.toml
├── data/global/state.json
├── data/agents/<agent_id>/memory.json
├── data/agents/<agent_id>/learning_log.jsonl
├── data/learning/events.jsonl
├── runs/
├── artifacts/
└── logs/
```

验收标准：

- 空目录执行后能得到完整 workspace。
- 重复执行不会覆盖已有 memory 和 runs。
- 缺失目录会补齐。

### 4.3 增加环境诊断命令

新增 CLI：

```bash
presentation-agent doctor --json
```

检查项：

- Python 版本。
- 当前 repo 是否存在。
- workspace 是否初始化。
- configs / skills 是否可读。
- 可选依赖是否存在，如 `python-docx`、`python-pptx`。
- 当前 git remote 和 commit。

输出示例：

```json
{
  "ok": true,
  "repo": "/Users/name/PresentationAgent/repo",
  "workspace": "/Users/name/PresentationAgent/workspaces/default",
  "python": "3.11.8",
  "checks": [
    {"name": "configs", "status": "ok"},
    {"name": "workspace", "status": "ok"}
  ]
}
```

验收标准：

- Agent 可以通过 JSON 判断是否需要安装、初始化或修复。
- 非技术用户不需要阅读复杂报错。

### 4.4 增加高层 report 命令

当前 `pipeline-init / step prepare / step commit / pipeline-advance` 对宿主 Agent 暴露了太多内部细节。

建议新增更高层命令：

```bash
presentation-agent report start --workspace <path> --brief-file brief.json
presentation-agent report next --workspace <path> --run <run_id_or_dir>
presentation-agent report submit --workspace <path> --run <run_id_or_dir> --output-file output.json
presentation-agent report approve --workspace <path> --run <run_id_or_dir>
presentation-agent report status --workspace <path> --run <run_id_or_dir>
```

职责：

- `report start`：初始化 pipeline，返回 run id 和当前 instruction。
- `report next`：返回当前阶段要做什么。
- `report submit`：提交宿主模型产物，执行 schema / review / stop check。
- `report approve`：阶段完成后由用户确认推进下一阶段。
- `report status`：查看整体状态和产物路径。

验收标准：

- Host Adapter Skill 只需要理解这 5 个命令。
- 内部 step / pipeline 目录结构可以后续变化，不影响外部 skill。

### 4.5 改造现有模块支持 workspace

需要改造的模块：

| 文件 | 改造点 |
|---|---|
| `presentation_agent/cli.py` | 所有命令增加 `--workspace` 或读取默认 workspace |
| `presentation_agent/loop.py` | MemoryStore / LearningEventStore / artifacts 输出改用 workspace 路径 |
| `presentation_agent/pipeline.py` | run_state 和 stage dir 写入 workspace/runs |
| `presentation_agent/memory.py` | 不再默认写 repo/data |
| `presentation_agent/learning.py` | learning events 写入 workspace/data/learning |
| `presentation_agent/web.py` | 后续可选支持 `--workspace` 启动 |

验收标准：

- 用临时 workspace 跑完整 pipeline，不污染 repo。
- 多个 workspace 可以并存。

## 5. 第二阶段执行方案：Skill + Git 分发

### 5.1 官方 GitHub 仓库作为分发源

目标：

```text
用户不手动下载代码。
Agent 根据 skill 自动 clone / pull 官方仓库。
```

默认安装目录：

```text
~/PresentationAgent/repo
```

默认 workspace：

```text
~/PresentationAgent/workspaces/default
```

### 5.2 增加安装/更新命令或脚本

建议新增 CLI：

```bash
presentation-agent install-local
presentation-agent update-local
```

也可以先由 skill 直接执行 git 命令。

MVP 安装流程：

```bash
mkdir -p ~/PresentationAgent
git clone https://github.com/<org>/<repo>.git ~/PresentationAgent/repo
cd ~/PresentationAgent/repo
python -m presentation_agent.cli init-workspace --workspace ~/PresentationAgent/workspaces/default
python -m presentation_agent.cli doctor --workspace ~/PresentationAgent/workspaces/default --json
```

MVP 更新流程：

```bash
cd ~/PresentationAgent/repo
git pull
python -m presentation_agent.cli init-workspace --workspace ~/PresentationAgent/workspaces/default
python -m presentation_agent.cli doctor --workspace ~/PresentationAgent/workspaces/default --json
```

验收标准：

- 安装后能跑 `doctor`。
- 更新不会覆盖 workspace。
- Agent 能根据 `doctor --json` 判断下一步。

### 5.3 改造 report_builder skill 为薄适配器

当前问题：

```text
HARNESS_ROOT=/Users/zhangsijing/Desktop/Coding/presentation_agent
```

需要改成：

```text
默认 repo: ~/PresentationAgent/repo
默认 workspace: ~/PresentationAgent/workspaces/default
```

新 skill 应只包含：

1. 何时触发。
2. 如何检查安装。
3. 如何安装 / 更新。
4. 如何把用户需求整理成 brief。
5. 如何调用高层 report 命令。
6. 如何记录 feedback / success memory。

不要包含：

- 开发者本机路径。
- 过长内部 step 细节。
- 复杂 Python 模块解释。

### 5.4 Host Adapter Skill 模板

建议目录：

```text
templates/host_adapters/
├── codex/SKILL.md
├── claude-code/SKILL.md
└── workbuddy/SKILL.md
```

三者内容大部分一致，只在平台特定安装位置和调用习惯上不同。

后续可增加：

```bash
presentation-agent install-adapter --host codex
presentation-agent install-adapter --host claude-code
presentation-agent install-adapter --host workbuddy
```

但 MVP 可以先手动提供三份模板，让 Agent 自己复制。

验收标准：

- 在 Codex 里能通过 skill 自动 clone repo。
- 在 Claude Code / WorkBuddy 里至少能按同样协议执行。
- skill 中没有本机绝对路径。

## 6. 第三阶段执行方案：MCP 工具化

这一阶段可以放在 CLI + Git 分发稳定之后。

### 6.1 MCP Server 能力

新增：

```bash
presentation-agent mcp serve --workspace <path>
```

暴露工具：

| 工具 | 作用 |
|---|---|
| `report_start` | 创建新汇报 run |
| `report_next` | 返回下一步 instruction |
| `report_submit` | 提交模型输出 |
| `report_approve` | 人工确认推进 |
| `report_feedback` | 记录 human review 反馈 |
| `memory_dream` | 触发 memory 整理 |
| `doctor` | 检查环境 |

验收标准：

- 支持 MCP 的 Agent 终端不用 shell 命令也能调度。
- 与 CLI 使用同一套底层 service。

### 6.2 与 CLI 共享服务层

不要让 MCP 重新实现逻辑。

建议抽出：

```text
presentation_agent/service.py
```

CLI 和 MCP 都调用 service。

```text
CLI command -> service -> runtime
MCP tool   -> service -> runtime
```

验收标准：

- CLI 和 MCP 行为一致。
- 修改 loop 协议时只改 service / runtime，不改多个入口。

## 7. 建议实施顺序

### Milestone 1：去路径硬编码

任务：

- 删除 `skills/report_builder/SKILL.md` 中硬编码 `HARNESS_ROOT`。
- 增加默认路径约定。
- 加 `doctor --json`。
- 加 `init-workspace`。

验收：

- clone 到任意目录都能运行。
- `doctor` 能返回清晰 JSON。

### Milestone 2：workspace 分离

任务：

- 增加 `presentation_agent/workspace.py`。
- Memory / runs / artifacts 改用 workspace。
- CLI 全部支持 `--workspace`。

验收：

- 用两个 workspace 跑不会互相污染。
- git pull repo 不会影响 workspace 数据。

### Milestone 3：高层 report 命令

任务：

- 新增 `report start / next / submit / approve / status`。
- 内部复用现有 pipeline-init / step prepare / commit / advance。

验收：

- Host skill 不再需要理解 stage_dir / pipeline_dir 细节。
- 能跑完一个完整 7-Agent 流程。

### Milestone 4：薄 Host Adapter Skill

任务：

- 重写 `skills/report_builder/SKILL.md`。
- 增加 `templates/host_adapters/`。
- 定义 Codex / Claude Code / WorkBuddy 统一触发协议。

验收：

- 用户说“安装汇报助手”，Agent 能自动 clone / init。
- 用户说“更新汇报助手”，Agent 能自动 pull / doctor。
- 用户说“做一份汇报”，Agent 能自动调用 report 命令推进。

### Milestone 5：GitHub 分发

任务：

- 清理 repo 中不应发布的本地文件。
- 完善 README：给人看的说明 + 给 Agent 看的安装协议。
- 发布 GitHub Pages 静态介绍页。

验收：

- 新用户只需要把 GitHub 地址告诉 Agent。
- Agent 能完成安装和第一次运行。

### Milestone 6：MCP 化

任务：

- 抽 service 层。
- 增加 MCP server。
- 更新 host adapter：优先 MCP，fallback CLI。

验收：

- 支持 MCP 的终端体验更顺滑。
- 不支持 MCP 的终端仍可用 CLI。

## 8. 当前最小可行方案

如果只做最快 MVP，建议只做这 4 件：

1. `init-workspace --workspace <path>`
2. `doctor --json`
3. 高层 `report start / next / submit / approve / status`
4. 重写 `report_builder` 为 GitHub clone + CLI 调度的薄 skill

这样就能实现：

```text
把 GitHub 仓库地址发给别人
→ 别人的 Agent 终端根据 skill 自动 clone
→ 初始化 workspace
→ 调用 report 命令
→ 生成汇报材料
```

## 9. 风险与注意事项

### 9.1 不要覆盖用户数据

禁止：

```text
git reset --hard workspace
删除 workspace/data
更新时覆盖 memory.json
```

必须：

```text
repo 可更新
workspace 只迁移不覆盖
memory append-only 或 merge
```

### 9.2 不要让 skill 变厚

skill 越厚，越难升级。

应该把复杂逻辑放在 CLI / service 层，skill 只负责：

- 触发；
- 安装；
- 调用；
- 呈现结果；
- 记录反馈。

### 9.3 非技术用户需要 Agent 可读说明

README 不只写给人，也要写给 Agent。

建议增加：

```text
AGENT_INSTALL.md
AGENT_USAGE.md
```

内容要包含明确命令、错误处理、何时询问用户。

## 10. 一句话总结

当前最佳路线不是马上做 PyPI 或 MCP，而是先做：

```text
GitHub repo 分发 + workspace 分离 + 薄 Host Adapter Skill + 高层 CLI
```

这样既能满足非技术用户“让 Agent 帮我安装和使用”的体验，也能保留后续产品化到 MCP / 插件市场 / 私有部署的空间。
