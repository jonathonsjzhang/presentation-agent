# 汇报助手流水线质量复盘与修改方案

> 基于 2026-07-01 "DS用户时长分析→元宝竞争启示" PPT 全流程复盘
> Run ID: `report-2026-07-01T094749Z0000`
> Pipeline: argument_synthesis → storyline_design → page_filling → format

---

## 一、复盘过程

用户对最终产出提出三类不足，经逐环节中间产物回溯（从最终 PPT 逆向到 argument_synthesis 的 evidence_bank，再正向比对原始 DOCX 材料），完成了全链路因果定位。关键发现：**三个问题虽表现为不同症状，但都指向 argument_synthesis 环节——它在第一步就完成了“过度归纳（丢证据）+ 强判断措辞（造强词）+ 编造 roadmap（造数字）”三连击。** storyline 忠实继承了这些问题，并在局部（P5“不可替代”）进一步加码。

## 二、三个问题的因果链

### 不足①：Leadline 判断过强，缺乏严谨性

**表现**：PPT 中出现“结构性逆转”“不可替代的高价值壁垒”“结构性引擎”等封闭式强判断词，但页面推理链不足以支撑。

**因果链追踪**：

```text
原始 DOCX 材料
  说法“DS时长增长呈极强线性趋势，增速明显高于豆包、元宝”
       ↓ [argument_synthesis]
  KA-01 claim 写为：“竞争格局正在发生结构性逆转”
  core_thesis 写为：“自我强化飞轮…形成竞争壁垒”
       ↓ [storyline，忠实继承]
  P2 leadline：“竞争格局正在发生结构性逆转”
  P5 leadline：加码为“不可替代的高价值壁垒”
       ↓ [page_filling → format]
  最终 PPT 呈现为强结论，但缺乏支持“结构性”“不可替代”的推理链
```

**根因**：

- argument_synthesis 的 invariants 只有“证据与主张强度匹配”这一抽象原则，没有可操作的 claim 类型和支持等级。
- 相关性证据、访谈机制线索和因果证据被等同使用。
- reviewer 只检查 leadline 是否为完整判断句，未检查判断强度与证据类型是否匹配。

### 不足②：论据严重丢失，说服力单薄

**表现**：原文 11 段访谈原话仅 3 段进入最终 PPT（命中率 27%）。丢失的关键信息包括：

- **用户访谈** 8 段：37 轮修改、30 万字无上下文限制、办公“离不开”、豆包竞品对比、恶搞试探→信任迁移的完整旅程、“不蠢就不换”等；
- **解释变量**：DS v4 能力提升（长文、长记忆、幻觉改善）；
- **caveat**：2 月 peak 是节日后共性、长假后均有相似规律。

**因果链追踪**：

```text
原始 DOCX 材料
  11段访谈原话 + 多个 caveat/解释变量
       ↓ [argument_synthesis]
  evidence_bank 把多段访谈打包压缩成 1 条 E14
  DS v4 / 长假共性 caveat 没有完整进入 evidence_bank
  evidence_gaps 错误地把“未充分使用的原文信息”标记为“数据缺口”
       ↓ [storyline → page_filling]
  下游只能反复引用少量 quote
       ↓ [最终PPT]
  结论密集，但证据层次与机制解释不足
```

**根因**：

- argument_synthesis 直接从“读材料”跳到“归纳结论”，没有先完整盘点、再选择性使用。
- evidence_bank 没有证据原子性和来源定位要求。
- 当前材料投影机制还可能只向 argument 提供长字段 preview，使“过度归纳”与“输入不完整”叠加。

### 不足③：漂浮的 roadmap

**表现**：PPT P9/P10 出现“2–4 周内完成能力缺口评估”“Q3 路线图中明确优先级”“3–6 个月实现时长拐点”等时间承诺，原材料中完全没有这些数字。

**因果链追踪**：

```text
原始 DOCX 材料
  没有“3–6个月”“2–4周”“Q3”“拐点”“缺口评估”“路线图”
       ↓ [argument_synthesis]
  decision_request 凭空生成 2–4周和 Q3
  why_now 凭空生成 3–6个月
  so_what 凭空生成时长拐点
       ↓ [storyline]
  leadline / closing_intent 忠实继承
       ↓ [page_filling]
  visual_plan 进一步丰富为时间轴
```

**根因**：

- “不编造数字”没有覆盖建议、目标和计划中的数字。
- schema 强制要求自由文本 `why_now`，会诱导模型为每个议题创造紧迫性。
- “action-oriented”没有和“execution planning”划清边界。
- 建议粒度只在局部提示词中出现，没有成为跨阶段项目级约束。

## 三、修改原则

本轮不只追加提示词，而是建立四层控制：

```text
原始材料完整进入上下文
  → Evidence Catalog 完整盘点
  → argument 对证据做有理由的取舍，并校准 claim 强度
  → storyline/page/format 不升级判断、不生成执行计划
  → schema + machine check + reviewer + E2E eval 分层拦截
```

核心原则：

1. **完整盘点不等于全部上屏**：系统必须知道所有证据，但只选择与决策问题相关、质量足够的证据进入主论证。
2. **判断强度来自证据类型，不来自文风偏好**：词表只是辅助，claim 类型与支持等级才是主约束。
3. **战略建议清晰但不过度执行化**：回答做不做、优先级、聚焦谁和什么场景，不自动生成排期、KPI、owner、预算或路线图。
4. **P0 必须可执行**：能机械判断的规则必须落入 schema 或 machine check，不能只写在自然语言 rubric 中。
5. **下游不替上游修错**：发现上游 overclaim 时阻断并回退，不静默继承，也不静默降级。

## 四、详细修改方案

### 改动 0：修复 argument 的材料可见性

**问题**：当前 `configs/context_requirements.json` 对 `argument_synthesis` 声明了 `materials / research_findings / complete_evidence`，但没有像 page filling 一样配置专项 inline limit，因此仍使用全局默认的单字段 12,000 字符和 3 条 preview。材料过长时，argument 可能只收到预览；如果 provider 不支持在生成中实际读取 `material_refs`，提示词里的“按需读取”并不能补回全文。

**施工内容**：

1. 为 `argument_synthesis` 和新增的 `evidence_harvester` 配置独立字段预算，至少覆盖：
   - `materials`
   - `research_findings`
   - `complete_evidence`
   - `paragraphs / raw_text`
   - `data_tables`
   - `images`
2. 在 `context_manifest.json` 中记录每个字段是 `full / preview / reference_only`。
3. Worker 开始生成前执行 input readiness gate：
   - 需要盘点的字段若只有 preview，且 provider 不能实际读取 artifact path，则 P0 阻断；
   - 图片、表格等模态未检查时标为 unresolved，不得声称“证据已完整盘点”。

**验收条件**：

- argument/evidence worker 使用的所有材料字段均可证明是 full inline 或已实际读取；
- 不允许把 preview 当完整材料；
- context manifest 可追溯每个源文件、字段和模态的读取状态。

### 改动 1：新增 Evidence Catalog，而不是把所有证据塞进 argument

#### 1.1 新增 Evidence Harvester

新增只读、专职取证的 Worker：`evidence_harvester`。它负责完整盘点，不负责形成战略结论。

采用“确定性拆分 + LLM 分类”的混合方案：

```text
Connector
  → 将 DOCX/PDF/XLSX 拆成带稳定 ID 的 source units
  → Evidence Harvester 对 source units 分类、解释和标注
  → deterministic coverage check 检查所有 source units 都有处置结果
```

Harvester 输出 `evidence_catalog.v1`：

```json
{
  "source_units": [
    {
      "source_unit_id": "DOC1-P037",
      "source_id": "DOC1",
      "source_location": "paragraph 37",
      "modality": "text",
      "raw_content": "完整原文",
      "inspection_status": "inspected"
    }
  ],
  "evidence_items": [
    {
      "evidence_id": "EV-001",
      "source_unit_refs": ["DOC1-P037"],
      "type": "quote",
      "raw_content": "完整原文",
      "normalized_observation": "不改变原意的结构化描述",
      "attribution": {
        "speaker": "访谈对象A",
        "profile": "年龄/职业等原文已有信息"
      },
      "scope": "individual_case",
      "limitations": ["单一访谈，仅用于机制侧证"]
    }
  ],
  "source_unit_disposition": {
    "DOC1-P037": {
      "status": "captured",
      "evidence_refs": ["EV-001"]
    }
  },
  "unresolved_units": []
}
```

证据类型至少覆盖：

- `quantitative_data`
- `quote`
- `case`
- `caveat`
- `counterevidence`
- `explanatory_variable`
- `author_judgment`
- `definition_or_methodology`

#### 1.2 完整盘点与论证取舍分离

Evidence Catalog 保存所有材料；argument 只选择与 decision goal 相关且质量足够的证据。不得要求所有访谈、所有解释变量都进入 key argument 或最终 PPT。

argument 新增 `evidence_disposition`：

```json
{
  "EV-001": {
    "status": "selected",
    "role": "mechanism_support",
    "claim_refs": ["KA-03"],
    "reason": "用于解释信任迁移机制，不外推总体"
  },
  "EV-002": {
    "status": "appendix",
    "role": "additional_case",
    "claim_refs": ["KA-03"],
    "reason": "与主线证据重复，保留供追问"
  },
  "EV-003": {
    "status": "excluded",
    "role": "irrelevant",
    "claim_refs": [],
    "reason": "与本次 decision goal 无关"
  }
}
```

`status` 枚举：

- `selected`：进入主论证；
- `counterpoint`：作为反证、替代解释或边界；
- `appendix`：相关但不进入主线；
- `excluded`：无关、重复或证据质量不足。

**语义边界**：

- `evidence_gaps` 只记录“为了证明某一 claim 但原材料中缺失的证据”；
- 未使用的已有材料不得写入 `evidence_gaps`；
- `evidence_usage_map` 不由模型重复生成，而由程序根据 `key_arguments[].evidence_refs` 和 `evidence_disposition` 派生，避免两套映射不一致。

#### 1.3 Harvester 动态触发

为保持 Manager 的动态 loop，不把 Evidence Harvester 固定为所有任务必跑：

- `deep_dive`、多源材料、包含访谈/问卷/大量定性研究时强制运行；
- quick sync 或短材料可由 argument 内部执行同 schema 的轻量 evidence pre-pass；
- 是否触发由 Manager planning 根据材料类型、数量和风险声明，不由 Worker 自行决定。

### 改动 2：结构化 claim 强度，而不是主要依赖禁词表

#### 2.1 argument schema 增加 claim 类型与支持等级

每个 `key_argument` 增加：

```json
{
  "id": "KA-01",
  "claim": "DS 用户时长呈持续增长趋势",
  "claim_type": "descriptive",
  "support_level": "directional",
  "logic_chain": {
    "observations": ["EV-001", "EV-002"],
    "inference": "剔除共性节日峰值后仍有增长信号",
    "implication": "该现象值得进入持续性和驱动因素分析"
  },
  "evidence_refs": ["EV-001", "EV-002"],
  "counterevidence_refs": ["EV-003"],
  "boundary_conditions": ["时间窗和数据口径待核验"],
  "confidence": "medium"
}
```

`claim_type`：

- `descriptive`
- `comparative`
- `explanatory`
- `causal`
- `predictive`
- `recommendation`

`support_level`：

- `supported`
- `directional`
- `hypothesis`

#### 2.2 判断校准规则

追加到 `skills/argument_synthesis/SKILL.md`：

```text
1. 相关性、交叉分析和访谈不能生成 causal claim；最多生成 explanatory hypothesis。
2. 单一访谈只能支持 individual case 或 mechanism clue，不得外推总体。
3. predictive claim 必须有时间序列、预测模型或原材料中明确的预测依据，否则降为 hypothesis。
4. recommendation 必须由前序 descriptive/comparative/explanatory claim 推导，不得直接从单条 evidence 跳到建议。
5. 原材料作者或受访者使用强词，不代表报告可以将其作为作者判断继承：
   - 可以作为带 attribution 的原话引用；
   - 若要成为报告 claim，仍需独立通过证据强度校准。
6. “显著”仅在材料提供统计显著性检验时使用；仅有较大差值时使用“明显/高于/差距为…”并展示具体数值。
7. “持续”必须有足够时间窗和稳定趋势支持；单点或短期变化不得使用。
```

#### 2.3 绝对词表作为辅助检查

保留高风险词表，但取消“只要原文用了就允许复用”的例外：

```text
结构性逆转、不可替代、必然、唯一、已经形成、不可逆转、
决定性、终局、全面领先、碾压、翻盘、确定性壁垒
```

命中后必须满足以下二者之一：

- 作为明确标注 attribution 的原文引用；
- claim_type、support_level 和 evidence_refs 能证明该强度，且 reviewer 明确放行。

### 改动 3：重构决策建议字段，消除 schema 对 roadmap 的诱导

#### 3.1 将 `why_now` 改为可选、有来源的 urgency

当前 `executive_summary.why_now` 是必填自由文本，会迫使模型为每个议题生成“现在必须行动”的理由。修改为：

```json
{
  "decision_relevance": "为什么该问题值得管理层讨论",
  "urgency_basis": {
    "statement": "仅在原材料存在明确时间窗口时填写",
    "source_refs": ["EV-010"]
  }
}
```

规则：

- `decision_relevance` 必填，但不表达虚构的时间紧迫性；
- `urgency_basis` 可为 `null`；
- 非空时必须包含 source refs；
- 没有材料依据时，不得出现“尽快、窗口收窄、时间不多、未来几个月”等暗示。

#### 3.2 将 recommendation 从自由文本改为战略决策结构

用结构化 `decision_request` 替代自由生成的 roadmap：

```json
{
  "decision_type": "priority",
  "recommended_direction": "优先验证办公与深度研究场景是否具备建立信任锚点的条件",
  "rationale_claim_refs": ["KA-02", "KA-03"],
  "specificity_level": "strategic_direction",
  "source_backed_specifics": []
}
```

`decision_type`：

- `do_or_not`
- `priority`
- `focus`
- `positioning`
- `tradeoff`
- `discussion`

`specificity_level`：

- `strategic_direction`
- `strategic_choice`
- `execution_plan`

本项目默认只允许前两级；`execution_plan` P0 阻断。

#### 3.3 决策建议边界

允许：

- 做/不做；
- 战略优先级；
- 聚焦人群、场景或能力方向；
- 差异化定位；
- 需要管理层确认的战略取舍；
- 需要补证后再决定的方向。

除非用户明确要求执行规划且材料提供依据，否则禁止：

- 周/月/季度/半年等排期；
- KPI、目标值、时长拐点、增长承诺；
- milestone、阶段路线图、甘特图；
- 负责人、组织调整、团队编制；
- 预算、人力和资源数量；
- “第一步/第二步/三阶段推进”等执行步骤；
- 缺乏来源的紧迫性和窗口期判断。

禁止范围覆盖：

- `core_thesis`
- `so_what`
- `expected_action`
- `decision_request`
- `why_now / urgency_basis`
- `closing_intent`
- storyline leadline
- page action item / visual plan

### 改动 4：Manager 将建议粒度作为项目级约束传播

在 `report_charter` 增加：

```json
{
  "recommendation_granularity": "strategic_direction",
  "unsupported_specificity_policy": "forbid",
  "evidence_inventory_policy": "full_catalog_for_deep_dive"
}
```

Manager task packet 必须继承这三个字段，并在验收 argument、storyline、page filling 和 format 时检查。

不要只在 argument task packet 中写自然语言约束，因为 roadmap 也可能由 storyline、page filling 或 format 再次生成。项目级结构化字段是单一事实源，四个阶段都必须遵守。

### 改动 5：storyline 不放大；发现上游 overclaim 时退回

新增 P0 rubric：

```json
{
  "id": "SL-CLAIM-CALIBRATION",
  "severity": "P0",
  "dimension": "claim_calibration",
  "criterion": "Leadline 不得升级上游 claim 的 claim_type、support_level、因果强度、确定性或建议粒度。",
  "check": "逐页对照 source_argument_refs；检查是否新增绝对判断、因果、预测、时间表、KPI 或执行步骤。",
  "fix": "恢复证据允许的强度；若上游 claim 本身 overclaim，向 Manager 提交 blocking upstream revision request，退回 argument 修复。"
}
```

同时解决现有规则冲突：

- Storyline 不应一边“必须忠实继承”，一边自行把上游 claim 降级；
- 上游不合格时，Storyline 应阻断并请求 Manager 返工 argument；
- storyline schema 增加 `upstream_revision_requests`，至少支持：
  - `overclaim`
  - `unsupported_causality`
  - `missing_evidence`
  - `recommendation_too_specific`

修改 `presentation_agent/review.py` 的 upstream signal check：

```text
原规则：
“上游尖锐判断被模糊化、稀释”视为退化。

改为：
“下游是否无依据升级或无理由弱化上游判断；若上游判断本身超过证据边界，
下游是否提交 upstream revision request，而不是静默继承或静默改写。”
```

### 改动 6：让 P0 真正可机械执行

仅在 `rubrics.json` 中增加自然语言 check，仍主要依赖 LLM reviewer，不是真正稳定的硬门。扩展 `presentation_agent/machine_check.py`，新增：

- `forbidden_pattern`：检查禁止的 roadmap/KPI/组织/资源表达；
- `cross_ref_exists`：检查 evidence/claim/source ref 是否真实存在；
- `conditional_required`：如 `urgency_basis != null` 时强制 `source_refs`；
- `set_coverage`：检查所有 source units 都有 disposition；
- `enum_max_level`：检查建议粒度不得超过 `strategic_choice`；
- `derived_ref_coverage`：根据 key arguments 自动生成 evidence usage，并核对 disposition。

新增硬门：

#### ARG-EVIDENCE-DISPOSITION P0

```text
evidence_catalog 中每条 evidence item 必须有 selected/counterpoint/appendix/excluded
之一的处置状态；selected/counterpoint 必须被至少一个 claim 引用。
```

#### ARG-CLAIM-GROUNDING P0

```text
所有 supported/directional claim 必须引用真实 evidence；
没有 evidence 的只能标为 hypothesis，且不得成为确定性 core thesis。
```

#### ARG-RECOMMENDATION-SCOPE P0

```text
specificity_level 不得为 execution_plan；
urgency、时间、KPI、owner、预算等具体项必须有明确 source provenance，
且必须是用户要求的输出范围。
```

#### SL/PAGE/FORMAT-NO-AMPLIFICATION P0

```text
下游不得新增上游不存在的 timeline、KPI、owner、预算、组织调整或路线图。
```

Reviewer 必须同时拿到当前 artifact、上游 artifact 和 evidence catalog 的必要索引；否则无法判断“是否来自原文”。

### 改动 7：Memory 保持现有闭环，不作为安全门

当前 Worker loop 已在每轮 review 后调用 `record_objections()`，无需新增 `on_violation: record_to_memory` 解析逻辑。

本次只修正文档和使用原则：

- P0 硬规则直接进入 schema/rubric/machine check，不依赖 memory 召回；
- memory 用于发现尚未固化的新型错误和场景偏好；
- `promotion_candidates` 只产生候选；
- promotion 必须人工确认；
- 当前实现晋升为 P1 rubric，不会自动晋升为 P0。

### 改动 8：重构回归测试与 E2E 指标

不再使用“访谈 80% 必须上屏”或“所有数字清零”作为主要目标。前者会诱导堆素材，后者会误伤有来源的数字。

| Case ID | 测试目标 | 输入特征 | 通过条件 |
|---|---|---|---|
| REGR-E01 | Evidence inventory recall | 人工标注访谈、数据、caveat、解释变量 | quote/caveat/explanatory variable 召回率 ≥95%，无虚构条目 |
| REGR-E02 | Evidence disposition | catalog 含相关、重复、反证和无关素材 | 100% evidence item 有合法 disposition，未使用证据不进入 evidence_gaps |
| REGR-A01 | Claim grounding | 混合定量、相关性和单访谈证据 | 100% supported/directional claim 有有效 ref；访谈不外推总体 |
| REGR-A02 | Claim calibration | 只有趋势和相关性，无因果证据 | 不生成 causal/supported claim；强词为 0 或仅存在于 attribution quote |
| REGR-R01 | Unsupported roadmap | 原文没有排期/KPI/owner/预算 | 最终全链路无 unsupported execution specifics |
| REGR-R02 | Source-backed urgency | 原文明确存在监管日期或业务期限 | urgency 可保留，但必须带正确 source refs |
| REGR-D01 | Downstream amplification | 上游为 directional/hypothesis | storyline/page/format 不升级为确定性、因果或执行计划 |
| REGR-C01 | Context completeness | 超过 inline limit 的长材料与图片 | 未完整读取时 P0 阻断，不允许生成“完整盘点” |

同时保留词表扫描作为 cheap check，但最终质量使用旧/新版本 pairwise 盲评：

- 严谨客观程度；
- 论据厚度与证据多样性；
- 观点—证据—推理—边界闭环；
- 建议是否清晰但不过度执行化；
- 管理层可决策性；
- 是否存在材料外新增事实或计划。

## 五、优先级与实施路径

### 第一批：先消除最直接的过度生成与强判断

1. 修改 argument schema：
   - `why_now` 改为 `decision_relevance + optional urgency_basis`；
   - decision request 结构化；
   - key argument 增加 `claim_type / support_level`。
2. 更新 argument SKILL 与 rubrics：
   - claim 校准规则；
   - recommendation 粒度边界；
   - 原文强词不自动继承。
3. 在 report charter 增加项目级 recommendation granularity。
4. 修正 reviewer 的“尖锐判断弱化=退化”逻辑。
5. 为 storyline/page/format 增加 no-amplification P0。

这一批不依赖 Evidence Harvester，即可明显减少绝对化措辞和漂浮 roadmap。

### 第二批：解决证据完整性根因

1. 修复 argument 的 context projection 和 input readiness gate。
2. 升级 connectors，生成稳定 source unit IDs，并记录文本/表格/图片读取状态。
3. 新增 Evidence Harvester 和 `evidence_catalog.v1`。
4. 新增 `evidence_disposition`，区分完整盘点、论证取舍和附录保留。
5. 让 Manager 按 report type 和材料复杂度动态调度 Harvester。

### 第三批：把提示词约束升级为稳定硬门

1. 扩展 machine check 的 cross-ref、coverage、conditional 和 forbidden pattern 能力。
2. 实现 ARG-EVIDENCE-DISPOSITION、ARG-CLAIM-GROUNDING、ARG-RECOMMENDATION-SCOPE。
3. 实现跨阶段 SL/PAGE/FORMAT-NO-AMPLIFICATION。
4. 确保 reviewer 获得 evidence catalog 索引和完整上游信号。

### 第四批：回归评测与基准固化

1. 用本次 DS 材料建立人工标注 golden set。
2. 增加长材料、图片未读、原文有真实 deadline 等对抗用例。
3. 对旧版/新版做 pairwise 盲评。
4. 达标后再调整 E2E rubric 权重和 CI/nightly 阈值。

## 六、预期效果验证

修改后重新跑一遍同一份材料（Run ID: `report-2026-07-01T094749Z0000`），预期如下。

### 证据盘点层

- 11 段访谈均进入 Evidence Catalog，保留原文、归属和 source location；
- v4 能力变化、长假共性等解释变量和 caveat 被完整识别；
- 每条 source unit 都有 captured/excluded/unresolved 状态；
- 图片或表格未检查时明确阻断，不假装完整。

### Argument 层

- 每条核心 claim 都有 `claim_type / support_level / evidence_refs / boundary_conditions`；
- 访谈用于机制说明和案例，不外推为总体因果；
- 相关性证据不再生成“结构性逆转”“不可替代壁垒”等终局判断；
- 原材料中的 caveat 映射到受影响 claim；
- 未选入主论证的访谈进入 appendix 或 excluded disposition，而不是被误报为 evidence gap。

### 建议层

- 仍能明确回答做/不做、优先级、聚焦何种人群或场景；
- 不出现材料外生成的“2–4 周、Q3、3–6 个月、时长拐点”等执行计划；
- 没有材料依据时 `urgency_basis = null`，不虚构窗口期；
- 不生成 KPI、owner、预算、组织调整或阶段路线图。

### 下游层

- storyline leadline 不升级 claim 的因果性、确定性和建议粒度；
- 若上游 overclaim，storyline 发出 blocking revision request 并退回 argument；
- page filling 和 format 不再“丰富化”出 timeline、action item 或时间轴；
- 最终页面围绕关键观点形成“判断—证据—推理—边界”，不以访谈上屏率代替说服力。

### 量化验收

- Evidence Catalog 人工标注召回率：≥95%；
- Evidence disposition 覆盖率：100%；
- supported/directional claim 有效引用率：100%；
- unsupported execution specifics：0；
- 下游新增 unsupported claim/roadmap：0；
- 新版在严谨性、证据厚度和建议粒度三个 pairwise 维度均优于旧版。
