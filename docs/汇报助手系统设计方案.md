# 汇报助手（Presentation Assistant）系统设计方案

> 版本：v1.7（蓝图）｜更新：2026-06-17
> 定位：把"互联网战略分析师做高层汇报"的全流程，构造成一套由 7 个同构 Agent 串联、每个 Agent 内部以 Loop + State + Sub-agent 自我学习的系统。

---

## 0. 一句话概述

> 汇报助手是一条 **7-Agent 流水线**：`任务定位 → 核心论点提炼 → storyline 设计 → 单页内容填充 → format → Q&A 梳理 → 逐字稿`。每个环节都是一个**独立的 Agent**，内部是一套 **同构的半自动 Loop**——基于各自专属的 skill 编排工作流，由独立 sub-agent 做 maker/checker 审查与终止判定，最终由人拍板放行；运行中产生的反馈沉淀为冷数据 log，提炼成热更新的环节 memory 回流指导生成，形成自我进化闭环。

---

## 1. 系统概览：loop / skill / state 的关系

整个系统是一条 **7 个独立 Agent 串成的流水线**（任务定位 / 核心论点提炼 / storyline 设计 / 单页填充 / format / Q&A / 逐字稿），**每个环节都是一个独立的 Agent**，自带专属的 skill 与 memory。而每个 Agent 内部，又都是由 loop / skill / state 三类构件咬合而成的同构结构。先定位三者，再看它们如何协作。

| 构件 | 角色 | 一句话 |
|---|---|---|
| **loop** | 运行骨架 | 把单个 Agent 的一次生成跑成"发起 → 执行 → 审查 → 判定 → 人工 review"的闭环回路 |
| **skill** | 执行内核 | 定义 loop 中"执行"那一步的具体工作流（SOP）、硬约束、取数 connector、交接 schema |
| **state** | 记忆与约束 | loop 运行所依赖、并在运行中被更新的数据；分**全局 state**（跨 Agent 共享）与**环节 memory / log**（本 Agent 专属） |

### 三者如何咬合（放大单个 Agent，看一次运行的数据流）

```
   ┌─────────────────────────── 单个环节 Agent（如 ③ storyline 设计）───────────────────────────┐
   │                                                                                            │
   │              ┌──────────────────── state ────────────────────┐                            │
   │              │  全局 state：跨 Agent 共享（受众 / 偏好 / ExecSummary / 目标 action）        │
   │              │  环节 memory：本 Agent 专属，热更新（含本环节的风格偏好 / 历史反馈）          │
   │              └────────────────┬──────────────────┬──────────┘                            │
   │                         读取①  │                  ▲ 回写④                                  │
   │                                ▼                  │                                        │
   │  发起 ──► workflow 执行 ──► 审查 sub-agent ──► 终止判定 ──► 人工 review ──►│ 产物交付下一 Agent
   │          （由本 Agent 的            │                                  │                     │
   │            skill 编排：           P0/P1 异议                       人的反馈                  │
   │            SOP+硬约束+              │                                  │                     │
   │            connector+schema）      └────── 写入 learning-log（冷）③ ◄──┘                    │
   │                                                  │                                          │
   │                                      提炼/更新 环节 memory（热）②                            │
   │  └──────────────────────────── 这一整条就是该 Agent 的 loop ────────────────────────────┘  │
   └────────────────────────────────────────────────────────────────────────────────────────────┘
                                              │ 产物（按 schema）
                                              ▼
                                    ④ 单页填充 Agent（同构结构，再来一遍）
```

**读这张图的三条主线：**

1. **每个环节是独立 Agent，内部是同一套 loop**：图中放大的是 ③ storyline 设计这一个 Agent；其余 6 个环节各是一个结构相同、但 skill 与 memory 各自独立的 Agent。loop 串起发起→执行→审查→判定→人工 review，其中"执行"这一步**完全由本 Agent 的 skill 驱动**。
2. **skill 在执行时消费 state**：生成前读取①「全局 state（这次汇报的受众/偏好/目标 action，所有 Agent 共享）」+「本环节 memory（该 Agent 历史沉淀的偏好与反馈）」作为约束。
3. **loop 在结束时反哺本环节 state**：审查 sub-agent 的异议与人工 review 的反馈 → 写入③ learning-log（冷，不进上下文）→ 提炼② 成环节 memory（热更新）→ 下次运行回写④ 进生成。这条"生成→反馈→沉淀→再生成"就是单个 Agent 的自我学习闭环；7 个 Agent 各跑各的闭环，互不串味。

> 一句话：**系统 = 7 个独立 Agent 的流水线；每个 Agent 内部，skill 是 loop 的"执行内核"，state（全局 + 环节）是 loop 的"记忆与约束"，loop 把两者跑成一个会自我进化的闭环。** 后续第 3 节展开 loop（回路）、第 4 节展开 skill（workflow）、第 5 节展开 state、第 6 节展开学习闭环。

### 总基调：人在环内的半自动系统

这是 **human-in-the-loop 的半自动系统**，不是无人值守的全自动机器人——汇报材料的质量（标题是否成结论、故事线是否有起落）是主观判断，没有 `npm test` 那样的客观判分器。因此：**机器负责把回路跑顺（取数、生成、自检、审查、返工、清除 P0），但"完没完、敢不敢用"由人最终决定**，每个 Agent 的终止环节保留一个不可跳过的「人工 review」节点。

## 2. 系统两层结构

### 第一层（宏观）：7-Agent 流水线

```
① 任务定位 → ② 核心论点提炼 → ③ storyline 设计 → ④ 单页内容填充 → ⑤ format → ⑥ Q&A 梳理 → ⑦ 逐字稿
```

- 每个 Agent 自带专属的 skill + 环节 memory，互不共享。
- 前一个 Agent 的产物即后一个 Agent 的输入，交接格式由**接口契约 schema** 约定——schema 缝进相邻 skill 的 SOP 首尾（见第 4 节）。
- 一层**全局 state** 横向贯穿全部 7 个 Agent，承载跨 Agent 的共享约束（见第 5 节）。
- 流水线为**单向直线**：不设下游打回上游的回退机制，也不设统一编排层，由人手动推进各 Agent（见第 8 节）。
- ③ storyline 设计作为先行打磨的样板 Agent，跑通后其同构结构复制到其余 6 个。

### 第二层（微观）：单个 Agent 的同构 Loop

把任意一个 Agent 放大，内部都是第 1 节概览所示的同一套结构——loop（回路）驱动 skill（workflow）消费并反哺 state。下面三节分别展开 loop（第 3 节）、skill（第 4 节）、state（第 5 节）。

---

## 3. 回路层（主流程）

```
发起 → workflow 执行 → 审查 sub-agent → 终止判定 → 人工 review →（放行至下一 Agent）
```

| 环节 | 职责 | 性质 |
|---|---|---|
| 发起 | 人工触发，或上游 Agent 跑完后自动接续 | 人工/脚本接入点 |
| **workflow 执行** | 基于本 Agent 的 skill 编排工作流；环节由该 skill 自定 | AI harness |
| **审查 sub-agent** | 干净上下文审查产物，输出 P0/P1 异议；有 P0 自动返工 ≤2 轮 | AI harness |
| **终止判定** | 独立 checker 核对该 skill 定义的可验证条件是否全过且无 P0 | AI harness |
| 人工 review | 人拍板：可否流向下一 Agent | 人工接入点 |

### 关键分工关系

1. **workflow 从属于 skill**：工作流分几步、每步做什么、用哪些约束，全部由该 Agent 的 skill 定义；不同 Agent 环节不同，方案层面不固化。
2. **生成与审查上下文隔离（maker/checker split）**：审查 sub-agent 拿不到生成时的思路，只拿到 skill 的约束 + memory + 产物，避免生成者对自己产物的偏袒。
3. **审查 ≠ 终止判定**：审查 sub-agent 管质量挑刺（输出 P0/P1）；终止判定是另一个干净上下文的 checker，只核对可验证条件是否达标。二者均贯彻 maker/checker 分离。

### P0 / P1 异议分级

| 级别 | 定义 | 处置 |
|---|---|---|
| **P0** | 违反该 skill 定义的硬约束之一，或其它可机械判定的硬伤（如数据无来源标注） | 自动返工，上限 ≤2 轮，仍有 P0 则停下交人 |
| **P1** | 主观质量问题（钩子不够、节奏平、记忆点弱等） | 提示但不强制返工，交人工 review 判断 |

> P0 的具体条目因 skill 而异，由各 skill 给出可验证的硬约束清单；方案层面只要求"必须存在一组可机械判定的硬约束"。

## 4. workflow 内部：由 skill 定义的工作流

**workflow 是 skill 的运行态，不是系统层面写死的固定步骤。** 不同 Agent（任务定位 / 核心论点提炼 / storyline / 单页填充 / format / Q&A / 逐字稿）面对的问题不同，工作流分几步、每步做什么、收口要核哪些约束，**都由各自的 skill 决定**。本节只描述"任意 Agent 的工作流都应具备的通用形态"，不规定具体步骤数。

### 通用形态（每个 skill 各自具体化）

| 通用阶段 | 含义（具体环节由 skill 自定） |
|---|---|
| 输入准备 | **按上游接口契约（输入 schema）读取上游 artifact**；判断输入形态、补齐生成所需素材；含原始数据时先提炼成可用论据（见下文 connector）。 |
| 核心生成 | 该 Agent 的主体工作（如 storyline 的谋篇、format 的排版、Q&A 的问答梳理），由 skill 编排为它认为合适的若干步。生成时按维度取相关 memory 簇作软约束（见 5.5）。 |
| 收口自检 | 对照该 skill 定义的硬约束清单 + 全量扫 memory 触发词（命中后顺横向关联补查，见 5.5），命中则修正；**按本环节接口契约（输出 schema）产出结构化 artifact 交给下游**。 |

> 原则：让 skill 把工作流拆成"每步只关注一件事"的形态，让质量原则在执行中自然发生，而非事后对照长 checklist。拆成几步由 skill 决定，系统不强制。

### 接口契约缝进 SOP（不单设系统级设施）

相邻 Agent 的交接 schema **不是浮在系统层的独立模块，而是缝进每个 skill 的 SOP 首尾**，成为单环节 SOP 的固定两端：

- **读端（输入 schema）= SOP 第一步**：下游 skill 在"输入准备"阶段按约定 schema 解析上游 artifact，按字段精确读取，不靠猜。
- **写端（输出 schema）= SOP 最后一步**：上游 skill 在"收口自检"后按约定 schema 产出结构化 artifact 才算交付。
- 一份接缝 schema 由相邻两 skill 共享：上游的"输出 schema" ≡ 下游的"输入 schema"。例 ③ storyline → ④ 单页填充：

  ```
  [{ 页码, 标题, 关键问题, 论据清单, so what, 标记(主线/附录) }, ...]
  ```

> 系统层只保留一条原则：**相邻 Agent 必须以约定 schema 交接**；schema 的具体字段沉淀在各 skill 的 SOP 里，随 skill 演进，不在系统层集中维护。

### 改进1：connector 取数能力（内嵌于 skill）

connector 不是独立 harness，而是 skill 内部的取数能力层，被工作流的"输入准备"阶段调用。

- 适用：输入含原始数据（Excel/CSV/数据库导出）时，把高频取数沉淀为可复用接口（如 QM 数据拉取、财报解析、Excel 趋势计算模板）。
- 收益：从"现场写脚本"变成"调接口 + 填参数"，可复用接口越积越多。
- 风险最低、收益最直接，建议优先落地。

### 改进2：多候选并行择优（默认关的开关）

适用于"同一输入存在多种合理解法"的 Agent（storyline 尤为典型，同一素材可从竞争格局 / 用户行为 / 商业价值等角度切入）：

- 默认关闭（单条生成，省 token），用户可显式开启。
- 开启后并行产出 2-3 个不同切入角度的候选，各自隔离上下文。
- 候选 ≤3 封顶（受人的审查带宽限制）。
- 择优交给审查 sub-agent 横向对比 + 人最终选。

## 5. 状态层（State）

State 的设计目标是一对矛盾的平衡：**既要能随反馈动态更新，又不能因条目堆积导致生成时上下文臃乱、注意力漂移。** 解法是把"记录"和"生效"彻底分开——冷数据只增不读、热数据精简且按需取。

### 5.1 两类 state：全局 vs 环节

state 分两类，**机制完全相同**（都是 learning-log + memory 两层，见 5.2），区别只在**关注内容**与**作用范围**：

| | 作用范围 | focus 的内容 | 初始化 / 维护 |
|---|---|---|---|
| **全局 state** | 横向贯穿全部 7 个 Agent | 跨环节通用：受众画像、汇报偏好（数据驱动 / 忌花哨）、Executive Summary、目标 action、整体篇幅上限 | 由 ① 任务定位初始化；① 之后原则上稳定，需修订时由人显式更新，中间 Agent 不擅改 |
| **环节 state** | 仅本环节 Agent 专属，互相隔离 | 本环节特有：如 storyline 的 Wording / Leadline 偏好；format 的排版偏好；Q&A 的提问角度偏好 | 由本环节运行中的反馈持续沉淀（见第 6 节闭环） |

> 关系：生成时全局 state 与本环节 memory **并存读取**，全局是上层约束、环节是本环节细则；环节不覆盖全局。

### 5.2 统一设计：learning-log（冷）+ memory（热）两层

无论全局还是环节，每一套 state 都由两层构成。这是整个 state 方案的核心——**用冷热分层把"可追溯"与"少漂移"两个诉求解耦**：

| 层 | 角色 | 是否进生成上下文 | 类比 |
|---|---|---|---|
| **learning-log** | 原始档案：忠实记录每一次反馈 | **否**，只增不读，仅供追溯与提炼 | 案发记录 |
| **memory** | 可执行规则：从多条 log 提炼的精简条目 | **是**，但按需取（见 5.5） | 判例法条 |

> rubrics 不在 state 内——它是各 skill 内置的稳定硬约束，作为环节 memory 的晋升归宿（见第 4、6 节）。

### 5.3 learning-log 模板（冷数据）

log 一事一条、只追加。字段设计的关键是让它**日后可被聚类提炼**，因此在"场景·问题·原因·修改方式"之外，补了维度标签和关联指针：

```
## L-007
- 日期: 2026-06-17
- 所属: 环节(storyline) | 全局
- 维度: Wording | Leadline | 结构 | 证据 | 受众适配 | ...
- 触发场景: 在哪个环节、哪类产物上出的问题
- 问题: 用户原话或产物原样（例：标题写成“某产品是大盘唯一翻倍的产品”）
- 原因: 为什么算问题（例：绝对化措辞，高层易反感且易被举反例驳倒）
- 修改方式: 实际怎么改的（例：改为"少数实现翻倍的产品之一"）
- 关联: L-003, L-005（同维度的旧 log，供提炼时聚类）
```

`维度` 与 `关联` 是后续提炼成 memory、以及 memory 间建立关联的抓手。

### 5.4 memory 结构与分类（热数据，借鉴知识图谱的"原子 + 链接"）

memory 是真正进上下文生效的部分，必须精简。借鉴知识图谱思路，但**只借手法、不照搬知识库形态**（我们存的是"风格反馈"而非"世界知识"，必须比知识库更克制）。

**单条 memory 结构（原子化——一条只管一个主题）：**

```
- id: M-012
- 维度: Wording
- trigger: 标题/正文出现"极其 | 唯一 | 真正 | 只靠 | 必然 | 彻底"等绝对化词
  · 类型: 词面型（可机械扫词）
- suggestion: 换成有分寸的判断（"显著领先""少数之一"）
- case anchors: L-007, L-012（反指支撑它的 log）
- 关联: M-015（"标题别写成咨询腔"，同属 Wording 簇）
- 命中数: 4 ｜ 最近触发: 2026-06-17
```

**分类——按维度成簇**（以 storyline 环节为例）：

| 维度簇 | 管什么 | 示例 memory |
|---|---|---|
| **Wording**（措辞） | 用词分寸 | 别用绝对化词；标题别写成咨询腔；少用空泛形容词 |
| **Leadline**（标题/结论） | 标题是否成结论 | 这页只在陈述事实、没有 so what；标题不是完整句 |
| 结构 / 证据 / 受众适配 … | 各环节按需扩展 | —— |

**两类链接（借鉴双向链接）：**
- **纵向 `case anchors`**：memory → log，让 memory 极简的同时可追溯"为什么有这条规则"。
- **横向 `关联`**：memory ↔ memory，把同维度的相关条目连成簇（如"绝对化"关联"咨询腔"）。横向链接的用途见 5.5——**只在自检时顺链，不在生成时顺链**。

### 5.5 注入策略与防漂移（state 方案的命门）

先明确：所谓"注入"，就是**把 memory 库里相关的几条规则，作为文字写进当次生成的提示里**，让模型生成时就遵守。不是特殊机制，就是"把对的规则、在对的时机、贴进上下文"。难点只在——贴多了会漂移，贴少了会漏。解法是把生成与自检**分时、分量**对待。

#### 生成阶段：动笔前注入一小段"风格须知"

发生在第 4 节 workflow「核心生成」启动**之前**（模型真正写标题/谋篇之前），三步：

1. **判断这一步在做什么** → 反查涉及的维度。
   例：storyline 的核心生成是"起标题 + 排故事线" → 涉及 `Leadline`（标题/结论）+ `Wording`（措辞）两个维度。
2. **只取这几个维度簇的 memory**，且**不顺横向 `关联` 链往外扩**（只要本簇，不要邻簇）。
3. **把取出条目的 `suggestion` 拼成几行"风格须知"**，附在生成提示里。例如：

   ```
   【本次生成请遵守的风格须知】（历史沉淀）
   - 标题用有分寸的判断，不用"极其/唯一/真正/只靠"等绝对化词
   - 标题写成完整陈述句且带 so what，不要写成"现象：描述"的咨询腔
   - 少用"具有重要意义"等空泛形容词，换成具体可量化的结论
   ```

> **为什么这么克制**：生成阶段模型在做创造性工作，贴一长串零散禁忌会让它起标题畏手畏脚、趋于平庸。所以只带**少量、强相关、正向**的须知，宁可漏也不抢注意力——漏的部分由自检兜。

#### 自检阶段：全量扫 + 命中才顺链

发生在「收口自检」，此时创作已结束、纯机械核对，可以放开量：

1. **全量扫**整库 memory 的 trigger（词面型直接扫词，语义型逐条判断），不再限维度——正好补上生成时"维度没预判到"的漏。
2. **某条命中后，才顺它的横向 `关联` 链**带出邻近条目一起核对（如扫中"绝对化"，顺链把"咨询腔"也一并查），命中即按 `suggestion` 修正。

> 这条规则也回答了"横向 `关联` 链到底有什么用"——它的价值落在**自检补漏**（顺链查得更全），而**不是生成时塞更多**（那才会漂移）。

#### 一句话对比

| 时机 | 取哪些 | 怎么用 | 量 |
|---|---|---|---|
| 生成阶段 | 仅本步可预判维度的簇，不顺链 | suggestion 拼成"风格须知"贴进提示 | 少而精 |
| 自检阶段 | 全量 trigger + 命中后顺横向链 | 扫到即按 suggestion 修正 | 放开扫 |

#### 防膨胀（借鉴 lint）

memory 库设软上限（约 30 条/套）；定期清理——长期不触发的淘汰、近义条目合并、断了 anchor 的孤立条目修复。某条命中 ≥3 次则晋升为 skill 的 rubric 并从库删除（见第 6 节）。

## 6. 自我学习闭环

> 以下闭环在**单个环节 Agent 内部**发生，7 个 Agent 各跑各的、互不串味；全局 state 的闭环同理，只是反馈来源是跨环节的通用偏好。

```
① 写入：审查 sub-agent 异议 + 人工 review 反馈  ──→  learning-log（冷，按 5.3 模板）
② 提炼：相似 log（同维度、互相 关联）聚类      ──→  提炼/更新 memory（热，原子条目）
③ 回流：memory 按维度取簇随生成读取            ──→  指导 workflow
                                               （命中 ≥3 次 ──→ 晋升 rubric）
```

- **即时维护**：收到新反馈时，先按 5.3 模板写一条 log；再当场判断现有 memory 有无可覆盖的 trigger——有则把本 log 挂到该 memory 的 `case anchors` 并计数 +1，无则新建一条原子 memory（标注维度、必要时与同维度旧条目互建 `关联`）。
- **晋升通道**：某条 memory 命中 ≥3 次即视为稳定偏好，经用户确认后晋升进 skill 的 rubrics，并从 memory 库删除（连同其孤立的横向 `关联` 一并清理）。
- **防膨胀（lint）**：memory 库设软上限（约 30 条/套），定期清理——淘汰长期不触发、合并近义、修复断 anchor 的孤立条目。
- 结果：rubrics 保持精简，memory 承接长尾，log 沉底可追溯，三者按频次自然流动。

## 7. 明确不做的事

| 不做 | 理由 |
|---|---|
| **Automations（自驱心跳）** | 汇报是人发起的一次性创作，无需定时自动生成；强行自驱违背"终止阀门由人掌握"的总原则。 |
| **单独的"防认知投降关卡"** | 该职能已由人工 review 承担（人决定能否放行/上台），单设清单冗余。 |
| **流水线回退机制** | 保持单向直线。下游若发现上游产物撑不起来，不设自动"打回上游重做"（否则直线变成带反馈的网、复杂度陡增）。质量由各 Agent 自己的审查 sub-agent + 人工 review 在本环节内兜住；确需返工时由人手动回到对应 Agent 重跑。 |
| **统一编排层（orchestrator）** | 不设总控调度。由人手动逐个推进 7 个 Agent（发起节点支持"人工触发／上游自动接续"）。人本就是天然的编排者与终止阀门，额外编排层属过度设计。 |
