汇报助手系统框架介绍

Manager + 5 Worker Loop Harness for strategic reporting

System Framework

汇报助手是一套面向战略汇报生产的 Manager + 5 Worker Loop Harness。

它把材料从任务定义、分析、故事线、正式写作、追问清单到格式化拆成可控环节；每个环节由 skill 定义行为，由 review 拦截风险，由 state/memory 持续学习。

1+5Manager + Worker
5步单 Agent loop
3类 skill 契约文件
2种接入方式
01 · 系统总框架

主链路是五个默认 Worker；每个 Worker 由 loop 调度，并把 review 结果回流为 memory。

系统不是一次性生成整份材料，而是将汇报生产拆成多个专业环节。每个环节都有明确输入、明确产物、明确审查标准和人工放行点。

Trigger 人发起汇报任务 输入汇报对象、汇报性质、材料格式、素材、模板与历史参考。
Production Spine Manager + 5 Worker 汇报流水线 让 5 个专业 Worker 依次产出、审查、交接，而不是让一个模型一次性完成全部材料。
Analysis Storyline Report Q&A Format
Loop Scheduler 每个 Agent 都被同一套 loop 调度 start → workflow → sub-agent review → human review；P0 未清除则回到 workflow revise。
Runtime Start 启动与状态推进 标准化输入，读取配置，生成 instruction，推进 run_state。
Skill-based Workflow 基于 Skill Package 执行 SKILL.md 定义工作方式，rubrics 定义质量边界，schemas 保证上下游交接。
Review Gate Sub-agent & Human Review 机器先按 schema、rubric、memory scan 拦截硬伤，人再判断材料是否可用。
Self Improvement State & Memory 回流到下一轮生成 review 结果、人工反馈、成功案例与版本对比进入 learning events，提炼为 memory，并在下一轮由 retrieval / routing policy 影响生成与检查重点。
Observation Layer Web Cockpit 只是观察与编辑 harness 的可视化挂件，不是主生产链路。
02 · 5-Worker Pipeline

把“做一份汇报”拆成 5 个稳定、可审查、可替换的专业环节。

01Analysis把原始材料转成可追溯发现、解释、反证和 so what。产物：analysis.v1
02Storyline选择核心答案并组织递进章节，不重做分析。产物：storyline.v3
03Report写成完整、连续、可独立阅读的 Markdown 报告。产物：report.v1
04Q&A 梳理在报告末尾追加听众可能提出的深度问题，不写答案。产物：report.v1
05Format按目标载体选择视觉并由 runtime 渲染为正式材料。产物：formatted_material.v2

核心定义集中在 configs/agents.json：每个 Agent 声明输入输出 schema、上下游交接、memory 维度、rubrics 摘要、global state 读写范围、review policy 与可选能力。

03 · 单 Agent Loop

每个 Agent 内部都采用同构 loop，让产物先被机器拦截，再交给人判断。

Agent Loop maker + checker 同一个结构复用于 Manager 与 5 个 Worker
01start接收人或上游 artifact
02workflow按 skill 生成产物
03reviewschema / rubric / memory scan
04stop_checkP0 清除才放行
05human_review人决定 approve / revise / stop
Machine Gate P0 未清除 自动回到 workflow revise，不进入人工 review。
revise clear
Human Gate 人判断材料是否可用 approve 进入下一 Agent；revise 可回本环节或上游。
Runtime 模块 loop.py 跑单 Agent maker-checker loop；pipeline.py 串起 5-Worker 流水线；step.py 支持宿主模型单步执行。
Skill 模块 skills/<agent_id>/ 定义角色、SOP、rubrics 与 schema。修改 Agent 行为时优先改 skill 包。
Review / Stop Check review.py 与 machine_check.py 负责 schema gate、机械检查、rubric 问题和 memory scan。
Human-in-the-loop 机器不假装全自动判断主观质量；系统清除硬伤，人最终判断材料能不能用。
04 · Skill、Connectors 与 Renderers

Skill 定义“怎么思考”，Connectors 负责“怎么取数”，Renderers 负责“怎么交付”。

skills/<agent_id>/SKILL.md 定义 Agent 的角色、工作流 SOP、输入理解方式、输出要求和注意事项。 skills/<agent_id>/rubrics.json 定义 P0/P1 审查标准，是 review 的核心质量边界。 skills/<agent_id>/schemas/*.json 定义输入/输出 JSON 结构契约，保证上下游交接稳定。
Connectors 取数层 presentation_agent/connectors/ 支持 docx、xlsx、csv，把原始素材转成 Agent 可消费的结构化输入。
Renderers 输出层 presentation_agent/renderers/ 支持 PPT、HTML、Docx，把 JSON artifact 转成可交付材料。
PPT 专业能力 presentation_agent/vendor/mck_ppt/ 提供更专业的布局、风格常量、deck builder 与 QA 能力。
05 · State / Memory / Learning

系统学习不靠把所有历史塞进 prompt，而是靠轻记忆、重证据、强检索、弱路由。

feedback人工反馈与 review 问题
success被认可的成功产物
comparisonv1 到 vN 的差异反思
event logappend-only 记录事实
hot memory提炼短经验卡
dreaming定期合并、去重、清理矛盾
retrieval生成前召回少量相关记忆
routing policy调整检查重点和严格度
06 · 接入方式

系统既可以作为 Agent 终端里的 skill 使用，也可以通过 runtime adapter 接入不同模型或执行环境。

产品入口 / 使用方式 通过 skill 接入 Agent 终端 skills/report_builder/SKILL.md 把整个 harness 封装成 Codex、Claude Code、WorkBuddy 可调用的 report-builder skill。
底层执行抽象 通过 LLM Adapter / Host Integration 接入 presentation_agent/llm/ 提供统一 LLMClient、factory，以及 mock、cli、inline adapters。
