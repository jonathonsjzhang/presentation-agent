汇报助手系统框架介绍

一套面向战略汇报生产的 Manager + Worker Loop Harness：Manager 定义任务、规划、派发和验收，六个专业 Worker 负责论点、故事线、单页、格式、Q&A 与逐字稿。

01 · 系统总框架

这不是一个单一脚本，而是一套 loop-first 的战略汇报生产系统。

系统把汇报生产拆成明确输入、明确产物、明确审查标准和明确人工放行点的环节。模型负责生成与修订，harness 负责装配指令、校验结构、推进流程、记录状态与沉淀经验。

Trigger 人发起汇报任务 输入汇报对象、汇报性质、材料格式、素材、模板与历史参考。
Production Spine Manager 控制面 + 专业 Worker Manager 围绕项目目标动态定义、派发和验收任务，Worker 在隔离上下文中执行。
Manager 论点提炼 Storyline 单页草稿 Format Q&A 逐字稿
Loop Scheduler Manager loop 调度 Worker loop planning → dispatch → Worker review → Manager acceptance；必要时返工或向用户升级。
Runtime Start 启动与状态推进 标准化输入，读取配置，生成 instruction，推进 run_state。
Skill-based Workflow 基于 Skill Package 执行 SKILL.md 定义工作方式，rubrics 定义质量边界，schemas 保证上下游交接。
Review Gate Sub-agent & Human Review 机器先按 schema、rubric、memory scan 拦截硬伤，人再判断材料是否可用。
Self Improvement State & Memory 回流到下一轮生成 review 结果、人工反馈、成功案例与版本对比进入 learning events，提炼为 memory，并在下一轮由 retrieval / routing policy 影响生成与检查重点。
Observation Layer Web Cockpit 只是观察与编辑 harness 的可视化挂件，不是主生产链路。
02 · Manager + Workers

Manager 对项目结果负责，专业 Worker 对各自交付物负责。

M汇报项目 Manager定义 report charter、执行计划、任务包和验收标准。产物：Manager decision
01核心论点提炼提炼 Executive Summary、核心结论、关键论点和希望推动的 action。产物：Executive Summary
02Storyline 设计设计每页标题、关键问题、so what、证据安排，并通过标题连读测试。产物：Storyline
03单页内容填充把 storyline 展开成 dummy page、page brief 和图表信息。产物：Page content
04Format把结构化内容转成 PPT、HTML 或文档形态的正式材料。产物：Formatted material
05Q&A 梳理预判追问、风险点、回答策略和需要汇报人补充的信息。产物：Q&A pack
06逐字稿生成完整汇报话术、讲述节奏和时间分配。产物：Speaker script

核心定义集中在 configs/agents.json：每个 Agent 声明输入输出 schema、上下游交接、memory 维度、rubrics 摘要、global state 读写范围、review policy 与可选能力。

Worker 先完成专业自审，再由 Manager 按项目目标验收。

Human Gate Manager 判断任务是否通过 dispatch 后续任务、revise 返工，或 ask_human 升级。
3.1 Runtime 模块 manager.py 运行控制面；step.py 驱动 Worker maker-checker loop；pipeline.py 保留固定顺序兼容入口。
3.2 Skill 模块 skills/<agent_id>/ 定义角色、SOP、rubrics 与 schema。修改 Agent 行为时优先改 skill 包，而不是改 Python runtime。
3.3 Review / Stop Check review.py 与 machine_check.py 负责 schema gate、机械检查、rubric 问题和 memory scan。Stop check 只判断能否进入人工 review。
Human-in-the-loop 机器不假装全自动判断主观质量；系统清除硬伤，人最终判断材料能不能用。
3.2 · Skill、Connectors 与 Renderers

Skill 定义“怎么思考”，Connectors 负责“怎么取数”，Renderers 负责“怎么交付”。

skills/<agent_id>/SKILL.md 定义 Agent 的角色、工作流 SOP、输入理解方式、输出要求和注意事项。 skills/<agent_id>/rubrics.json 定义 P0/P1 审查标准，是 review 的核心质量边界。 skills/<agent_id>/schemas/*.json 定义输入/输出 JSON 结构契约，保证上下游交接稳定。
Connectors 取数层 presentation_agent/connectors/ 支持 docx、xlsx、csv，并由 registry 按文件后缀和 Agent spec 选择合适 connector，把原始素材转成 Agent 可消费的结构化输入。
Renderers 输出层 presentation_agent/renderers/ 支持 PPT、HTML、Docx，把 format / page filling 阶段的 JSON artifact 转成可交付材料。
PPT 专业能力 presentation_agent/vendor/mck_ppt/ 提供更专业的布局、风格常量、deck builder 与 QA 能力。
04 · State / Memory / Learning

系统学习不靠把所有历史塞进 prompt，而是靠轻记忆、重证据、强检索、弱路由。

feedback人工反馈与 review 问题
success被认可的成功产物
comparisonv1 到 vN 的差异反思
event logappend-only 记录事实
hot memory提炼短经验卡
dreaming定期合并、去重、清理矛盾
retrieval生成前召回少量相关记忆
routing policy调整检查重点和严格度
data/global/state.json 跨 Agent 共享约束，如受众、目标 action、Executive Summary、页数限制。 data/agents/<agent_id>/learning_log.jsonl 本 Agent 的冷数据日志，记录反馈、review 问题和修改方式。 data/agents/<agent_id>/memory.json 生成时可读取的短经验卡，只保留可执行、可复用的经验。 data/agents/<agent_id>/memory_summary.json 定期整理后的记忆摘要，用来降低重复和冲突。 data/learning/events.jsonl 项目级事件流，记录 feedback、success、comparison、retrieval 与 dreaming。
05 · 接入方式

系统既可以作为 Agent 终端里的 skill 使用，也可以通过 runtime adapter 接入不同模型或执行环境。

产品入口 / 使用方式 通过 skill 接入 Agent 终端 skills/report_builder/SKILL.md 把整个 harness 封装成 Codex、Claude Code、WorkBuddy 可调用的 report-builder skill。宿主按 next / submit / approve / feedback 执行当前 Manager 或 Worker 指令。
底层执行抽象 通过 LLM Adapter / Host Integration 接入 presentation_agent/llm/ 提供统一 LLMClient、factory，以及 mock、cli、inline adapters，让同一套 loop / skill / review / schema 跑在不同执行方式下。
接入方式面向对象主要入口适合场景
Skill 形式接入Codex / Claude Code / WorkBuddyskills/report_builder/SKILL.md人在对话里发起任务，宿主 Agent 调度 harness。
LLM Adapter 接入harness 内部 runtimepresentation_agent/llm/CLI、mock、inline 等不同执行模式统一模型调用。
